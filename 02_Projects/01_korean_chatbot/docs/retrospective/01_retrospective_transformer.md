# 세부 과제 5 - 한국어 챗봇 만들기

## 트러블 슈팅 1 - 모델 아키텍처 선택 (Encoder-Decoder vs Decoder-Only vs Encoder-Only)

### 문제 상황

- 기존에 배운 Transformer 구조는 Encoder + Decoder가 결합된 형태
- 그러나 GPT, LLaMA 등 생성형 모델은 대부분 Decoder-Only 구조를 사용함
- 챗봇 과제의 특성(입력 문장을 받아 이어서 생성하는 자동회귀 생성 모델)에 맞는 구조를 선택할 필요

### 고려한 옵션

| 구조              | 주요 용도                                   | 우리 과제 적합도 | 선택 여부 |
| ----------------- | ------------------------------------------- | ---------------- | --------- |
| Encoder + Decoder | 번역, 요약, Sequence-to-Sequence            | 보통             | 비선택    |
| Encoder-only      | 문장 이해, 분류, 검색 (BERT 스타일)         | 낮음             | 비선택    |
| **Decoder-only**  | 텍스트 생성, 챗봇, 언어 모델링 (GPT 스타일) | **높음**         | **선택**  |

### 결정 및 이유

- 최종 결정: Decoder-Only Transformer
- 선택 이유:
  - Decoder-Only 구조는 Causal Mask를 통해 자동회귀 생성을 자연스럽게 지원함.
  - 과제의 목적이 "문장을 입력받아 자연스럽게 이어서 생성"하는 것이기 때문에 Decoder-Only 구조가 적합

## 트러블 슈팅 2 - 구현의 깊이 수준 선택

### 문제 상황

- "가급적 PyTorch를 사용하지 않고 직접 만들어보는 것"이 해당 과제의 핵심
- 그러나 "직접 만들기"의 범위가 모호함
  - `nn.Transformer`, `nn.MultiheadAttention` 같은 "고수준 모듈"까지 사용하지 말아야 하는가?
  - `nn.Linear`, `nn.LayerNorm` 같은 기본 모듈까지 직접 구현해야 하는가?
  - Attention 메커니즘만 직접 구현하면 되는가?

### 고려한 옵션

| 옵션 | 내용                                                            | 장점                                | 단점                                                    | 평가                 |
| ---- | --------------------------------------------------------------- | ----------------------------------- | ------------------------------------------------------- | -------------------- |
| A    | `nn.Transformer`, `nn.MultiheadAttention` 모두 사용             | 가장 빠르게 구현 가능               | 학습 효과 낮음, 강사님 의도와 거리 있음                 | 비추천               |
| B ✅ | 기본 모듈(`nn.Linear` 등)은 사용하고, **Attention만 직접 구현** | 학습 효과와 시간 효율의 균형이 좋음 | `nn.MultiheadAttention`을 사용하지 않으므로 구현량 증가 | **선택**             |
| C    | `nn.Linear`까지 직접 구현                                       | 학습 효과 가장 높음                 | 시간 소모가 매우 큼                                     | 시간 부족으로 비추천 |
| D    | LSTM으로 먼저 전체 파이프라인을 만들고 Transformer로 교체       | 파이프라인 완성 속도가 빠름         | Transformer 직접 구현 경험이 약해짐                     | 비추천               |

### 결정 및 이유

- 최종 결정: 옵션 B
  - `nn.Linear`, `nn.LayerNorm`, `nn.Dropout` 등 기본 모듈은 사용하고,
  - **Attention 메커니즘(Scaled Dot-Product Attention, Multi-Head Attention)은 직접 구현**하는 방향으로 진행.

- 선택 이유
  - 알렉스가 강조한 "직접 만들기"의 핵심은 **고수준 완성형 모듈(`nn.MultiheadAttention`, `nn.Transformer`)을 블랙박스로 사용하지 않는 것**이라고 판단.
  - `nn.Linear`와 같은 단순한 기본 모듈까지 직접 구현하는 것은 학습 효과 대비 시간 비용이 크다고 판단.
  - 제한된 시간 내에 **동작하는 결과물**을 만들면서도, Attention 메커니즘을 직접 구현하는 학습 효과를 어느 정도 확보할 수 있는 가장 현실적인 선택이라고 판단.

- 하위 결정: `nn.Linear` 직접 구현 여부
  - `nn.Linear`와 같은 기본 모듈까지 직접 구현할지 여부를 별도로 고민
  - 다만, 학습 효과 대비 시간 비용이 크다고 판단
  - 최종적으로 `nn.Linear`는 사용하고, Attention 구현에 집중하기로 결정

### 인사이트

- “직접 만들기”의 범위를 무조건 극단적으로 해석하기보다는, **학습 목적과 시간 제약을 함께 고려**하여 현실적인 수준을 정하는 것이 실무에서도 중요할 것으로 예상

## 트러블 슈팅 4 - Causal Mask의 차원에 따른 코드 수정 위치 정립 (Single Head vs Multi-Head)

### 문제 상황

- `ScaledDotProductAttention` 테스트 중 Causal Mask를 적용했을 때, mask 차원에 따라 결과 shape이 깨지는 현상 발생
- 처음 AI가 제공한 테스트 코드에서는 테스트용 mask가 4차원으로 생성했으나, `scores`가 3차원이라 broadcasting이 실패하면서 shape이 비정상적으로 변형됨
- 이 과정에서 **"테스트 코드를 수정해야 하는지, 아니면 `ScaledDotProductAttention` 내부 코드를 수정해야 하는지"** 코드 수정 위치에 따른 "책임 분리 결정"에 대한 고민

### 원인 분석

- `ScaledDotProductAttention`의 `scores`는 `(batch, seq, seq)` 형태의 **3차원** 텐서
- Multi-Head Attention으로 확장하면 `scores`가 `(batch, heads, seq, seq)` 형태의 **4차원**이 됨
- 따라서 mask 차원도 단계에 따라 다르게 구성해야 함
- 차원 개수가 맞지 않으면 `masked_fill`에서 broadcasting이 깨져 shape이 비정상적으로 변형

### 결정 및 대응

- 최종 결정:
  - 테스트 코드를 수정하여 mask를 3차원으로 맞추는 방향으로 진행
  - `ScaledDotProductAttention` 내부에서 mask 차원을 자동으로 맞추는 로직은 추가하지 않음

### 결정 이유

- 현재 `ScaledDotProductAttention` 단계는 Mask의 결과가 **3차원** `(batch, seq, seq)` 또는 `(1, seq, seq)`으로 구성되는 것은 정상
- 이는 **현재 단계에서 테스트의 범주**이며 추후 Multi-Head 까지 확장하여 고려하면, 테스트 코드는 3차원, n차원 등 어떤 차원이든 동일한 결과가 나와야 하는 것도 설득력있는 설계 방법으로 고려할 수 있음.(예시: `shape: (batch_size, n_den, ...)`)
- 다만 지금은 Single Head이므로 테스트의 범주를 고려했을 때, `ScaledDotProductAttention` 내부에서 mask 차원을 자동으로 맞추는 로직은 과도한 일반화로 판단하여 현재는 추가하지 않음
- 대신 해당 설계 방법은 추후 확장 과정에서 의미있는 설계 아이디어이므로, `test_attention.py`에 **TODO 주석을 추가**하여 Multi-Head 구현 시 mask 차원 변경이 필요함을 명시

### 인사이트

- Attention 구현 시 mask 차원 관리가 매우 중요함
- mask 차원이 맞지 않을 때, **어디를 수정할 것인가**에 대한 판단 기준을 세우는 것을 고민
- Single Head와 Multi-Head 단계에서 mask 차원이 달라질 수 있으므로, 단계별로 mask 구성을 명확히 구분해야 함
- 차원 불일치로 인한 에러는 디버깅이 어렵기 때문에, 추후 작업으로 표시하고, 현재는 테스트 코드를 수정하는 단계로 변경

## 트러블 슈팅 5 - nn.Linear의 역할 이해 (Q, K, V Projection)

### 문제 상황

- `MultiHeadAttention` 구현 과정에서 `self.q_linear = nn.Linear(d_model, d_model)`과 같이 `nn.Linear`를 사용하는 코드를 작성함
- `nn.Linear`가 내부적으로 어떤 동작을 하는지, 왜 `in_features`와 `out_features`를 지정해야 하는지, 그리고 `forward` 시점에 실제로 어떤 일이 일어나는지에 대한 이해가 부족했음
- 특히 "입력 텐서의 마지막 차원 크기와 `in_features`가 일치해야 하는 이유"와 "`d_model, d_model`처럼 동일한 값을 넣는 의미"에 대해 혼란을 겪음

### 원인 분석

- `nn.Linear(in_features, out_features)`는 레이어를 **생성하는 시점**에 내부적으로 `(out_features, in_features)` 크기의 가중치 행렬 W와 bias를 생성함
- 이 가중치 행렬의 크기를 결정하기 위해 `in_features`와 `out_features`를 미리 지정해야 함
- 실제 선형 변환(`input @ W.T + bias`)은 `forward` 단계에서 이루어지며, 이때 입력 텐서의 마지막 차원 크기가 `in_features`와 정확히 일치해야 행렬 곱셈이 정상적으로 수행됨
- `d_model, d_model`처럼 입력과 출력 차원이 같을 경우에도, 단순히 차원을 유지하는 것이 아니라 **입력 정보를 특정 관점(Q, K, V)으로 재해석**하는 의미가 있음

### 결정 및 대응

- `nn.Linear`의 동작 원리를 다음과 같이 정리함:
  - 레이어 생성 시(`__init__`): 가중치 행렬 W 생성 및 변환 준비
  - 실제 사용 시(`forward`): 입력 텐서의 마지막 차원을 기준으로 선형 변환 수행
- `in_features`와 입력 텐서의 마지막 차원 크기가 일치해야 한다는 점을 명확히 인지
- `d_model, d_model`처럼 동일한 값을 사용하는 경우에도 **정보의 재해석**이라는 의미가 있음을 이해

### 인사이트

- `nn.Linear`는 단순히 차원을 늘리거나 줄이는 도구가 아니라, **학습 가능한 선형 변환**을 수행하는 레이어임
- 레이어를 생성하는 시점과 실제 데이터를 넣는 시점을 명확히 구분해야 함
- `in_features`는 입력 텐서의 마지막 차원 크기와 반드시 일치해야 하며, 이는 행렬 곱셈의 차원 조건 때문임
- `d_model, d_model`처럼 동일한 값을 사용하는 것은 차원을 유지하면서도 입력 정보를 특정 목적(Q, K, V)에 맞게 재해석하기 위한 설계임을 이해함

## 트러블 슈팅 6 - 폴더 구조에 따른 Import 에러 해결 (sys.path 수동 추가 방식)

### 문제 상황

- 프로젝트를 `src` 레이아웃으로 재구성한 후, `tests/` 폴더에서 모델 코드를 import하려고 했을 때 `ModuleNotFoundError: No module named 'src'` 에러가 발생함
- 예시 에러

  ```bash
  from src.model.decoder_layer import DecoderLayer
  ModuleNotFoundError: No module named 'src'
  ```

### 원인 분석

- Python은 기본적으로 sys.path에 등록된 경로만 모듈을 검색함
- `src` 폴더는 Python이 자동으로 인식하는 경로가 아니기 때문에, `from src.xxx import` 형태의 import가 실패함
- `src`를 패키지 루트로 인식할 필요

### 결정 및 대응

- 가장 기본적인 방법으로 테스트 파일 상단에 프로젝트 루트 경로를 동적으로 추가하는 방식 적용
- 코드 예시

  ```python
  import sys
  import os

  # 프로젝트 루트 경로 추가하여 임포트 에러 해결
  __TEST_FOLDER_PATH__ = os.path.dirname(__file__)
  joined_root_path = os.path.join(__TEST_FOLDER_PATH__, '..')
  absoluted_joined_root_path = os.path.abspath(joined_root_path)
  sys.path.append(absoluted_joined_root_path)
  ```

  - 이 코드를 통해 현재 테스트 파일의 위치를 기준으로 프로젝트 루트 경로를 계산한 후, sys.path에 추가하여 import가 가능하도록 함

### 인사이트

- sys.path.append 방식은 빠르게 문제를 해결할 수 있지만, 테스트 파일이 많아질수록 모든 파일에 동일한 코드를 중복해서 작성해야 한다는 단점이 있음
- 장기적으로는 유지보수성과 확장성이 떨어지는 방식이라는 점을 인지함

## 트러블 슈팅 7 - 확장 가능한 Import 해결을 위한 pyproject.toml 도입

### 문제 상황

- 프로젝트를 `src` 레이아웃으로 재구성하면서 `tests/` 폴더에서 모델 코드를 import할 때 `ModuleNotFoundError: No module named 'src'` 에러가 발생함
- 초기에는 테스트 파일 상단에 `sys.path.append`를 통해 프로젝트 루트 경로를 수동으로 추가하는 방식으로 문제를 해결했으나, 이 방식은 파일이 많아질수록 모든 테스트 파일에 동일한 코드를 반복적으로 작성해야 한다는 **유지보수 문제**를 안고 있었음
- 또한 `scripts/generate.py`처럼 별도 폴더에 있던 실행 스크립트에서 모델을 import할 때도 동일한 문제가 발생하여, 구조적으로 일관된 해결책이 필요하다고 판단

### 원인 분석

- `src` 레이아웃을 사용하면서 `pip install -e .`을 적용하지 않은 상태에서는 Python이 `src` 폴더를 패키지 루트로 인식하지 못함
- `sys.path.append` 방식은 단기적인 해결책일 뿐, 프로젝트 규모가 커질수록 기술 부채가 될 가능성이 높음
- 내부 파일들(`decoder_layer.py`, `transformer_model.py` 등)에서도 `from src.model...` 형태의 import가 남아 있어, `pip install -e .` 적용 후에도 충돌이 발생
- 실행 스크립트(`generate.py`, `main.py`)를 별도로 두고 import하려는 시도 또한 구조적으로 일관성을 해치는 요인이 되었음

### 결정 및 대응

- 장기적인 유지보수성과 확장성을 확보하기 위해 `pyproject.toml`을 도입하고, `pip install -e .`를 통해 프로젝트를 개발 모드로 설치하는 방식으로 전환
- `pyproject.toml` 소스 코드

  ```toml
  [build-system]
  requires = ["setuptools>=61.0"]
  build-backend = "setuptools.build_meta"

  [project]
  name = "korean-chatbot"
  ```

## 트러블 슈팅 8 - Pyright 타입 추론의 맥락 의존성 (Context-Sensitive Type Inference)

### 문제 상황

- `_merge_vocab()` 메서드 호출 여부에 따라 Pyright에서 타입 에러가 발생하거나 발생하지 않는 현상이 관찰됨
- 구체적으로 아래 코드

  ```python
  # bpe_tokenizer.py
  pairs = self._get_stats(vocab)
  ...
  best_pair = max(pairs, key=pairs.get)   # 여기서 타입 에러 발생 여부가 갈림
  vocab = self._merge_vocab(best_pair, vocab)
  ```

  - `_merge_vocab()` 호출 라인을 주석 처리하면 `max()`에서 타입 오버로드 에러가 발생하고, 주석을 해제하면 에러가 사라지는 현상이 발생함.

### 원인 분석

- 이 현상은 Pyright의 고급 타입 추론 방식 때문에 발생한다.
- Pyright는 단순히 한 줄씩 독립적으로 타입을 추론하는 것이 아니라, **함수 전체의 제어 흐름(Control Flow)**과 **데이터 흐름(Data Flow)**을 함께 분석한다. 특히 다음과 같은 특성을 가지고 있다:
  - Flow-sensitive Typing: 변수의 타입이 코드 흐름에 따라 좁혀지거나 변화할 수 있음
  - Context-sensitive Type Inference: 이후에 나오는 코드(사용 맥락)가 이전 코드의 타입 추론에 영향을 줄 수 있음
  - Bidirectional Type Inference: 타입 추론이 위에서 아래로만 진행되는 것이 아니라, 아래에서 위로도 영향을 줄 수 있음
- 결과적으로 `_merge_vocab()` 호출이 존재하면 Pyright가 `vocab` 변수의 타입을 더 구체적으로 추론하게 되고, 이로 인해 `pairs.get`의 반환 타입도 `int`로 더 확신 있게 판단하여 `max()` 호출에서 타입 에러가 발생하지 않았다.
- 반대로 해당 호출이 없으면 타입을 보수적으로 추론하여 `int | None`으로 판단하면서 에러가 발생한 것이다.

### 결정 및 대응

- 해당 현상이 런타임 에러가 아닌 정적 타입 체커(Pyright)의 동작 방식에 의한 것임을 인지
- 타입 에러를 해결하기 위해 아래 두 가지 방식 중 하나를 선택하여 적용:
  1. `max(pairs.items(), key=lambda item: item[1])[0]` 방식으로 변경 (타입 안전성 확보)
  2. `# type: ignore` 주석을 사용하여 타입 체커를 우회 (단기적 해결)

### 인사이트

- Pyright와 같은 현대적인 타입 체커는 단순한 타입 검사기를 넘어 코드의 맥락과 흐름을 이해하려고 시도한다는 것을 경험함
- `bidirectional type inference`, `context-sensitive inference`, `flow-sensitive typing`과 같은 개념이 실제 코드에서 어떻게 동작하는지 직접 확인할 수 있었음
- 타입 에러가 발생할 때 단순히 "타입이 안 맞는다"고만 생각하지 않고, **타입 체커가 코드를 어떻게 분석하고 있는지** 역으로 추론해보는 습관을 키울 수 있었음
- 앞으로 복잡한 타입 에러를 만났을 때, "이 에러가 발생하는 이유가 타입 체커의 추론 방식 때문일 수 있다"는 관점을 가질 수 있게 됨
- 타입 체커의 동작 원리를 깊이 이해하는 것이, 단순히 타입 에러를 피하는 것을 넘어 더 나은 코드 설계와 디버깅 능력으로 이어질 수 있다는 것을 확인

## 트러블 슈팅 9 - BPE train() 구현 과정에서 발생한 기술적 버그 수정

### 문제 상황

- BPETokenizer의 `train()` 메서드를 구현하는 과정에서, `token_to_id`에 `hello</w>, low</w>, lowe, newe` 등 비정상적인 토큰이 다수 포함되는 현상이 발생했다.
- `_get_stats()`와 `_merge_vocab()` 자체는 어느 정도 동작했으나, 여러 기술적 오류로 인해 학습 결과가 왜곡되었다.

### 원인 분석 &rarr; 결정 및 대응 &rarr; 결과

1. 초기 `word_freq` 생성 시 key 설정 오류
   - **동작되지 않았던 문제**: 초기 단어 데이터를 구성할 때 key를 잘못 지정하여, 문자 단위로 쪼갠 데이터가 제대로 저장되지 않았다.
   - **개선한 코드**:

     ```python
      # 수정 전
      word_freq[word] = word_freq.get(word_tokens, 0) + 1

      # 수정 후
      word_freq[word_tokens] = word_freq.get(word_tokens, 0) + 1
     ```

   - **결과**: 초기 데이터가 올바르게 구성되기 시작하면서, 이후 병합 과정이 의미 있는 방향으로 동작할 수 있는 기반이 마련되었다. (토크나이저의 데이터 초기화 정확도 개선)

2. `_merge_vocab()`에서 빈도 누적 방식 오류
   - **동작되지 않았던 문제**: 병합 후 동일한 형태로 수렴하는 단어들의 빈도가 덮어쓰여, 실제 빈도 정보가 손실되었다.
   - **개선한 코드**:

     ```python
      # 수정 전
      new_vocab[new_word] = freq

      # 수정 후
      new_vocab[new_word] = new_vocab.get(new_word, 0) + freq
     ```

   - **결과**: 병합 과정에서 빈도가 제대로 누적되기 시작하면서, 가장 빈번한 pair를 더 정확하게 판단할 수 있게 되었다. (토크나이저의 병합 정확도 개선)

3. 최종 `token_to_id` 구축 방식의 한계
   - **동작되지 않았던 문제**: `word_freq.keys()`만을 기준으로 토큰을 수집하여, 병합 과정에서 생성된 다양한 서브워드들이 제대로 반영되지 않았다.
   - **개선한 코드**:

     ```python
     for word in word_freq:
         all_tokens.update(word.split())

     # (추가) merge_rules에 등장한 토큰까지 수집
     for pair in self.merge_rules:
         all_tokens.add(''.join(pair))
     ```

   - **결과**: `merge_rules`에서 생성된 병합 토큰(`lo`, `we`, `st` 등)까지 `token_to_id`에 포함되면서, 전체 토큰 수가 증가하고 서브워드 단위의 다양성이 어느 정도 확보되었다. (토크나이저의 vocabulary 품질 개선)

### 인사이트

- BPE에서 `_merge_vocab`은 단순한 문자열 치환 함수가 아니라, 빈도를 정확히 유지하며 병합 결과를 반영하는 핵심 로직이라는 점을 깨달음
- 최종 `token_to_id`를 구성할 때는 현재 상태뿐만 아니라 병합 이력(`merge_rules`)까지 함께 고려해야 더 풍부한 서브워드 vocabulary를 만들 수 있음을 경험함
- `train()`의 여러 부분에서 작은 구현 오류가 누적되면서 전체 학습 결과가 크게 왜곡될 수 있다는 점을 확인함
- 코드 수정 시 “어떤 부분이 BPE의 본질적인 동작 원리와 맞지 않았는지”를 명확히 정의하고, 그에 맞춰 구체적인 로직을 개선하는 방식이 효과적임을 깨달음

## 트러블 슈팅 10 - BPE의 본질 이해 부족으로 인한 학습 로직 설계 한계

- 이 트러블 슈팅은 단순한 구현 오류가 아니라, **BPE의 본질을 이해하지 못한 상태에서 코드를 작성했던 개념적 한계를 기록한 내용**

### 문제 상황

- `train()` 메서드를 구현하면서 `_get_stats()`와 `_merge_vocab()`은 어느 정도 동작했으나, 전체적인 학습 구조가 BPE의 의도된 방향과 맞지 않아 의미 있는 vocabulary가 만들어지지 않았다.
- 특히 `token_to_id`에 전체 단어(`low</w>`, `hello</w>`)가 그대로 남거나, `lowe`, `newe` 같은 비정상적인 중간 형태의 토큰이 다수 포함되는 문제가 발생했다.

### 원인 분석

- BPE를 구현하면서 단순히 "pair를 합친다"는 구현 로직에만 집중한 것이 주요 원인이었다.
- BPE의 진짜 목적은 단순한 병합이 아니라, "**데이터를 압축하면서도 정보 손실을 최소화하는 방식으로 텍스트를 표현하는 것**"이다.
- 이 목적을 달성하기 위한 동작 원리는 **정보 이론(Information Theory)**에 기반한다.
  - Entropy: 자주 등장하는 패턴일수록 하나의 심볼로 표현하는 것이 정보적으로 효율적이다.
  - Minimum Description Length (MDL): 데이터를 가장 짧게 설명할 수 있는 표현이 가장 좋은 표현이다.
  - BPE는 이 원리를 **greedy**하게 근사하여, 가장 빈번한 pair부터 반복적으로 병합함으로써 전체 Description Length를 줄이는 방향으로 vocabulary를 구성한다.

- 그러나 초기 `train()` 구현에서는 이러한 이론적 배경을 제대로 이해하지 못한 채, `len(vocab)`을 기준으로 반복을 제어하거나, 최종 `token_to_id`를 `vocab.keys()`만으로 구성하는 등 BPE의 본질과 맞지 않는 구조로 설계했다.

### 결정 및 대응

- BPE의 목적과 동작 원리를 정보 이론 관점에서 재정립하고, 그에 맞춰 `train()`의 방향성을 수정했다.
  - **BPE가 빈도 기반 데이터 압축** 알고리즘임을 인식
  - `_merge_vocab()`에서 빈도 누적 로직을 적용
  - 최종 `token_to_id` 구축 시 `merge_rules`에 등장한 토큰까지 수집하는 방식으로 개선
  - `train()`의 반복 조건을 `len(vocab)` 대신 병합 횟수(`merges_needed`) 기반으로 변경

### 인사이트

- BPE는 단순한 문자열 병합 알고리즘이 아니라, **정보 이론**과 **데이터 압축의 원리**를 바탕으로 한 **subword tokenization 기법**이라는 점을 깨달음
- 구현에 앞서 "**왜 이런 방식으로 동작해야 하는가?**"에 대한 이론적 이해가 선행되어야, 올바른 구조로 코드를 설계할 수 있음
- `train()`에서 `len(vocab)`을 기준으로 반복을 제어하거나, 최종 vocabulary를 단순히 현재 상태에서만 추출하는 방식은 BPE의 본질과 맞지 않다는 것을 경험함
- 앞으로 subword tokenization 관련 알고리즘을 구현할 때는, `정보 이론적 배경(MDL, Entropy)`을 먼저 이해하고 접근하는 것이 중요하다는 교훈을 얻음
- 코드의 기술적 버그 수정도 중요하지만, `알고리즘의 철학과 목적`을 이해하는 것이 더 근본적인 해결책이 될 수 있음을 확인

## 트러블 슈팅 11 - BPE Tokenizer의 OOV 처리 방식 개선 (`<unk>` 토큰 도입)

### 문제 상황

BPETokenizer.encode() 메서드에서 학습되지 않은 토큰(OOV)이 발생할 경우 -1을 반환하도록 구현되어 있었다.

이로 인해 다음과 같은 문제가 발생했다:

- TransformerLanguageModel의 nn.Embedding 레이어에 -1이 입력되면서 IndexError: index out of range in self 에러가 발생
- FastAPI를 통한 실제 추론(generate()) 과정에서 파이프라인이 중단되는 현상
- OOV에 대한 명시적인 처리 전략이 없어 모델 입력의 안정성을 보장할 수 없음

### 원인 분석

기존 encode() 구현은 아래와 같이 OOV를 처리하고 있었다:

```python
if token in self.token_to_id:
    token_ids.append(self.token_to_id[token])
else:
    token_ids.append(-1)   # OOV 표시
```

이 방식은 다음과 같은 한계를 가지고 있었다:

- `-1`은 nn.Embedding이 허용하는 유효한 인덱스 범위가 아님
- OOV가 발생하는 이유(학습 데이터에 해당 subword가 없었음)에 대한 의미 있는 처리가 없음
- 모델 학습 및 추론 단계에서 일관된 입력을 보장할 수 없음

### 결정 및 대응

**최종 결정**: OOV 발생 시 -1 대신 명시적인 <unk> (unknown) 토큰으로 치환하는 방식으로 변경

**OOV 해결 4가지 방식**
| 방식 | 설명 | 장점 | 단점 | 단기 적용 적합도 | 선택 이유 |
| --- | --- | --- | --- | --- | --- |
| **`<unk>` 토큰 사용** | OOV 발생 시 미리 정의된 `<unk>` ID로 치환 | 가장 안정적이고 구현이 단순 | 의미 정보 손실 | ★★★★★ | **즉시 적용 가능 + 에러 방지 효과가 가장 큼** |
| Character-level Fallback | OOV 단어를 다시 문자 단위로 쪼개서 처리 | 정보 손실이 적음 | 구현 복잡도 높음 | ★★☆☆☆ | 구현 난이도가 높아 단기 적용 부적합 |
| 빈도 기반 필터링 | 드물게 나오는 subword는 vocabulary에 넣지 않음 | vocabulary 품질 향상 | OOV 발생률이 오히려 증가할 수 있음 | ★★★☆☆ | 단독으로는 OOV 문제를 근본 해결하기 어려움 |
| 학습 데이터 보강 | 학습 corpus를 더 다양하게 구성 | 근본적 해결 | 데이터 수집 및 학습 비용이 매우 큼 | ★★☆☆☆ | 단기간에 효과를 보기 어려움 |

`<unk>` 토큰을 선택한 핵심 이유

- 가장 시급한 문제 해결: `-1`로 인해 발생하는 `IndexError`를 즉시 제거할 수 있음
- 구현 난이도 대비 효과: 비교적 간단한 변경으로 모델 입력의 안정성을 크게 확보할 수 있음
- 실무 표준: 대부분의 NLP 프로젝트에서 OOV 처리의 첫 단계로 `<unk>` 토큰을 도입하는 것이 일반적
- 점진적 확장 가능: `<unk>` 토큰을 기반으로 이후에 빈도 필터링이나 Character-level Fallback을 추가하기에 용이함

**주요 변경 사항**

1. **`__init__` 수정**
   - `unk_token` 파라미터를 추가하여 유연하게 unk 토큰을 지정할 수 있도록 변경

   ```python
      def __init__(self, vocab_size: int, unk_token: str = "<unk>"):
      self.unk_token = unk_token
   ```

2. **train() 수정**
   - 학습 완료 후 vocabulary를 구축할 때 <unk> 토큰을 **ID 0으로 고정**하여 가장 먼저 등록
   - 나머지 subword 토큰들은 1부터 순차적으로 ID 부여
3. **encode() 수정**
   - OOV 발생 시 -1 대신 self.token_to_id[self.unk_token] (즉, ID 0)으로 치환하도록 변경
   - 더 이상 -1이 반환되지 않도록 함
4. **decode() 수정**
   - 알 수 없는 token ID가 들어올 경우에도 <unk> 문자열로 복원하도록 처리

### 인사이트

- OOV 처리는 단순히 에러를 방지하는 기술적 조치가 아니라, **모델의 입력 안정성**과 **학습 가능성**을 확보하는 중요한 설계 결정이다.
- 1처럼 의미 없는 값을 사용하는 것은 모델이 이해할 수 없는 입력을 만들어내기 때문에, 명시적인 특수 토큰(<unk>)을 사용하는 것이 훨씬 바람직하다.
- <unk> 토큰을 ID 0으로 고정하는 것은 디버깅 편의성과 일관성을 확보하는 실무적인 관례이다.
- BPE Tokenizer를 만들 때 **"학습되지 않은 입력을 어떻게 다룰 것인가"**에 대한 전략을 초기부터 명확히 정의하는 것이 중요하다는 점을 깨달음.
- 현재는 시급한 안정성 확보를 위해 단기적으로 <unk> 토큰 방식을 도입했으나, 이는 의미 정보 손실이라는 한계를 가지고 있다. 향후 해당 한계가 부각될 경우, 최소 빈도 필터링이나 Character-level Fallback과 같은 고도화된 OOV 처리 방식으로 점진적으로 확장할 계획이다.

## 트러블 슈팅 12 - model.generate() 메서드의 동작 원리 및 핵심 개념 이해

### 문제 상황

TransformerLanguageModel 클래스 내부에 구현된 generate() 메서드의 전체적인 동작 흐름과 각 기술적 요소(torch.no_grad(), [:, -1, :], temperature, top_k, torch.multinomial, torch.cat 등)의 의미를 명확히 이해하지 못했다.

특히 아래와 같은 질문들이 발생했다:

- generate()는 학습을 하는 함수인가?
- self(input_ids)[:, -1, :]는 왜 마지막 위치만 가져오는가?
- temperature와 top_k는 각각 어떤 역할을 하는가?
- torch.cat로 토큰을 붙이는 이유는 무엇인가?
- BOS 토큰과 EOS 토큰은 왜 필요한가?

### 원인 분석

generate() 메서드는 단순한 함수가 아니라, **Autoregressive(자동회귀) 텍스트 생성**의 핵심 로직을 담고 있다. 이 메서드 안에는 여러 가지 중요한 개념들이 복합적으로 사용되고 있었기 때문에, 각 요소를 개별적으로 이해하지 못하면 전체 흐름을 파악하기 어려웠다.

주요 혼동 지점은 다음과 같았다:

- generate()가 학습을 위한 함수인지, 추론을 위한 함수인지 구분이 명확하지 않음
- logits와 확률 분포의 관계, temperature가 확률 분포에 미치는 영향
- top_k 샘플링의 목적과 동작 방식
- 시퀀스(input_ids)에 새로운 토큰을 계속 붙이는 이유와 max_new_tokens의 의미

### 결정 및 대응

generate() 메서드의 전체 동작을 단계적으로 분석하고, 각 기술적 요소의 의미를 명확히 정리했다.

1. generate()의 본질
   - generate()는 **학습을 수행하는 함수가 아니다**.
   - @torch.no_grad()가 적용되어 있어 gradient 계산이 비활성화되며, 가중치 업데이트가 일어나지 않는다.
   - 이 함수의 역할은 **이미 학습된(또는 초기화된) 모델을 사용해 텍스트를 자동으로 생성**하는 것이다.

2. Autoregressive 생성 구조
   - generate()는 **반복적으로 다음 토큰을 예측하고 붙이는** 방식으로 동작한다.
   - self(input_ids)[:, -1, :]를 통해 마지막 위치의 logits만 추출한 후, 다음 토큰을 하나 예측한다.
   - torch.cat([input_ids, next_token], dim=1)을 통해 예측된 토큰을 기존 시퀀스 뒤에 계속 붙인다.
   - 이 과정을 max_new_tokens 횟수만큼 반복하여 문장을 점진적으로 생성한다.

3. 샘플링 기법 이해
   - **temperature**: 확률 분포의 뾰족함을 조절하여 생성의 다양성을 제어한다. 낮을수록 보수적이고 안정적인 생성, 높을수록 다양하고 창의적인 생성이 가능하다.
   - **top_k**: 상위 k개의 토큰만 후보로 제한하여, 낮은 확률의 이상한 토큰이 선택되는 것을 방지한다.
   - **torch.multinomial**: softmax를 통해 얻은 확률 분포에서 실제로 다음 토큰을 랜덤하게 샘플링하는 역할을 한다.

4. BOS / EOS 토큰의 역할
   - **BOS 토큰**: 문장의 시작을 모델에게 알려주는 특수 토큰. 현재 generate()에서는 자동으로 붙여주지 않는다.
   - **EOS 토큰**: 문장의 끝을 의미하며, 생성을 조기에 중단(Early Stopping)하는 데 사용될 수 있다. 현재 generate()에는 EOS 기반 중단 로직이 구현되어 있지 않아 max_new_tokens만큼 무조건 생성한다.

### 인사이트

- generate() 메서드는 단순한 반복문이 아니라, **Autoregressive 생성의 전형적인 패턴**을 보여주는 중요한 구현체이다. 각 기술적 요소(temperature, top_k, multinomial 등)는 생성 품질을 제어하기 위한 샘플링 전략의 일부이다.
- torch.no_grad()의 사용은 이 함수가 **추론 전용**임을 명확히 보여주며, 학습과 추론의 경계를 명확히 구분하는 것이 중요하다는 점을 다시 한 번 확인했다.
- 현재 generate()는 기본적인 생성 기능만 구현되어 있으며, 실제 서비스 수준에서 사용하기 위해서는 **EOS 기반 Early Stopping**, **repetition_penalty**, **Top-p sampling** 등의 고급 기능이 추가적으로 필요하다.
- 개념을 하나씩 뜯어보면서 이해하는 과정이, 단순히 코드를 읽는 것보다 훨씬 깊이 있는 학습이 된다는 점을 경험했다.

## 트러블 슈팅 13 - 테스트 결과 해석 및 1단계/2단계 목표 구분

### 문제 상황

test_generate_text()를 실행한 결과, 생성된 문장에 <unk> 토큰이 다수 포함되어 있었다.

예시 결과:

```bash
hello <unk><unk>w <unk><unk><unk><unk>you toweatheneis met lowest t weathegoworworl<unk>yoweat
```

이 결과를 보고 “테스트가 실패한 것인가?”, “<unk>가 나오는 게 문제인가?”에 대한 판단이 명확하지 않았다. 또한, 1단계 목표와 2단계 목표를 구분하는 기준이 모호해졌다.

### 원인 분석

현재 generate()의 결과가 좋지 않은 주요 원인은 다음과 같다:

1. **모델이 학습되지 않은 상태** (random weights)
2. **BPE Tokenizer의 vocabulary가 매우 작음** (dummy_corpus 사용)

이로 인해 많은 토큰이 OOV 처리되어 <unk>로 치환되고, 모델이 의미 있는 다음 토큰을 예측하지 못하는 현상이 발생했다.

문제는 이 현상을 **버그**로 볼 것인지, **현재 단계에서는 정상적인 현상**으로 볼 것인지에 대한 판단 기준이 부족했다는 점이다.

### 결정 및 대응

**1. 테스트 결과에 대한 판단 기준 수립**

- 현재 test_generate_text()의 assert 조건은 isinstance(result, str)와 len(result) > 0뿐이다.
- 이 테스트는 **구조적 연결과 기본 동작 검증**을 목적으로 하며, **생성 품질**을 검증하는 테스트가 아니다.
- 따라서 <unk>가 많이 나오는 것은 **현재 단계에서는 예상된 현상**이며, 테스트 실패로 간주하지 않기로 결정했다.

**2. 1단계와 2단계 목표의 명확한 구분**
| 단계 | 목표 | 포함 여부 | 비고 |
| --- | --- | --- | --- |
| **1단계** | OOV 에러 없이 동작 + generate() 흐름 검증 | 포함 | 구조적 연결 중심 |
| **2단계** | random weights → 학습된 weights로 전환 | 포함 | 의미 있는 생성 확인 |
| **1단계 마무리 작업** | 디버그 프린트 정리, 코드 정리, 테스트 보강 | 선택 | 핵심 목표 달성 후 진행 가능 |

- **결론**:
  - 1단계의 목표는 **구조적인 연결과 기본 동작 확인**이다.
  - “의미 있는 문장이 생성되는가”는 1단계 목표가 아니라 **2단계 목표**에 해당한다.
  - 따라서 현재 테스트 결과는 1단계 목표를 벗어나지 않으며, 2단계로 넘어가야 할 시점으로 판단했다.

**3. 테스트의 목적에 따른 역할 분담**

- test_bpe_with_transformer(): BPE + Model을 **직접 연결**해서 사용하는 경우를 테스트
- test_dependency_injection(): generate() 함수가 **의존성 주입**을 지원하는지 검증
- test_generate_text(): 전체 파이프라인(generate() 함수 호출)이 에러 없이 동작하는지 검증

각 테스트가 검증하는 대상이 다르다는 점을 명확히 구분했다.

### 인사이트

- 테스트를 작성할 때는 **무엇을 검증하려고 하는지**를 명확히 정의하는 것이 중요하다. 단순히 “실행이 되는가”를 확인하는 테스트와 “품질이 좋은가”를 확인하는 테스트는 목적이 완전히 다르다.
- 현재 단계에서는 **구조적 안정성 확보**가 우선이며, 생성 품질은 학습을 진행한 이후에 평가하는 것이 합리적이다.
- <unk> 토큰이 많이 나오는 현상은 OOV 처리가 잘 되고 있다는 증거일 수도 있지만, 동시에 vocabulary 품질과 모델 학습이 아직 부족하다는 신호이기도 하다.
- 1단계와 2단계의 목표를 명확히 구분하지 않으면, “언제까지 구조를 다듬어야 하는가”에 대한 판단이 흐려질 수 있다. 단계별 목표를 문서화하고 공유하는 것이 중요하다는 점을 다시 한 번 확인했다.

## 트러블 슈팅 14 - TransformerLanguageModel에 train() 메서드를 구현해야 하는가?

### 문제 상황

scripts/train.py에서 학습 루프를 작성하던 중, TransformerLanguageModel 클래스 안에 별도의 train() 메서드를 구현하지 않은 것이 문제되지 않을까 하는 의문이 들었다.

일반적으로 일부 코드에서는 모델 클래스 내부에 학습 로직을 포함시키는 경우도 있기 때문에, 현재처럼 학습 루프를 train.py에 별도로 작성하는 방식이 올바른 설계인지 판단이 서지 않았다.

### 원인 분석

이러한 의문은 PyTorch의 설계 철학과 nn.Module의 역할에 대한 이해가 충분하지 않았기 때문에 발생했다.

주요 혼동 지점은 다음과 같았다:

- 모델 클래스 안에 학습 관련 메서드(train(), fit() 등)를 넣어야 하는 것이 아닌가?
- model.train()을 호출하는 것과 별도의 train() 메서드를 만드는 것의 차이
- 학습 로직을 모델 클래스에 포함시키는 방식과, 별도 스크립트에 작성하는 방식 중 어느 것이 더 바람직한가

### 결정 및 대응

**1. PyTorch의 기본 설계 철학 수용**

nn.Module을 상속받은 모델 클래스의 주된 역할은 **모델 구조를 정의**하는 것이다. 구체적으로는 다음과 같다:

- **init**(): 레이어 및 파라미터 정의
- forward(): 순전파 로직 정의

반면, **학습 루프**(데이터 로딩, 손실 계산, backward(), optimizer step 등)는 모델 클래스 외부에서 작성하는 것이 PyTorch의 권장 방식이다.

따라서 TransformerLanguageModel 클래스 안에 별도의 train() 메서드를 구현할 필요가 없다고 결론지었다.

**2. model.train() vs 별도 train() 메서드**

- model.train(): nn.Module에서 기본으로 제공하는 메서드로, 모델을 **학습 모드**로 전환하는 역할을 한다. (Dropout, BatchNorm 등의 동작을 제어)
- 별도 train() 메서드: 학습 루프 전체를 모델 클래스 안에 넣는 방식으로, 이는 **필수가 아니며** 오히려 모델 클래스를 비대하게 만들 수 있다.

**3. 학습 로직을 분리하는 방식의 장점**

학습 루프를 scripts/train.py와 같은 별도 파일에 작성하는 것이 다음과 같은 이점을 가진다고 판단했다:

- **모듈화**: 모델 구조와 학습 로직이 분리되어 책임이 명확해짐
- **테스트 용이성**: 모델과 학습 과정을 독립적으로 테스트할 수 있음
- **유연성**: 다양한 학습 전략(예: 학습률 스케줄링, gradient accumulation 등)을 쉽게 적용 가능
- **가독성**: 모델 클래스가 과도하게 커지지 않음

### 인사이트

- PyTorch에서 모델 클래스는 **구조 정의**에 집중하고, 학습 로직은 별도 스크립트에서 관리하는 것이 일반적인 설계 패턴이다.
- model.train()과 model.eval()은 이미 nn.Module에서 제공하는 메서드이므로, 별도로 구현할 필요가 없다.
- 학습 로직을 모델 안에 넣는 방식은 일부 간단한 실험 코드에서는 사용되기도 하지만, 프로젝트 규모가 커지거나 유지보수가 필요한 경우에는 오히려 단점이 될 수 있다.
- "무엇을 모델 클래스 안에 넣을 것인가"에 대한 기준을 세우는 것이 중요하다는 점을 깨달았다. 모델의 **본질적인 역할**(순전파)과 **부가적인 역할**(학습 루프)을 구분하는 습관이 필요하다.

## 트러블 슈팅 16 - 한국어 BPE Tokenizer 학습 시도 및 한계 인식

### 문제 상황

2단계에서 영어 데이터로 최소 학습 루프를 성공적으로 구현한 후, 3단계로 넘어가 한국어 데이터로 BPE Tokenizer를 재학습해보았다.

목표는 한국어에 적합한 vocabulary를 구축하여 OOV를 줄이고, 이후 모델 학습 시 더 나은 생성 품질을 확보하는 것이었다. 그러나 여러 차례 데이터 규모를 확대(79문장 → 283문장)하며 시도했음에도 불구하고, 생성 결과가 크게 개선되지 않았다.

### 원인 분석

**1. 데이터 규모의 한계**

- 초기 79문장, 이후 283문장까지 확대했으나, 한국어의 다양성을 충분히 커버하기에는 여전히 부족했다.
- 영어의 경우 15문장으로도 어느 정도 패턴 학습이 가능했던 것과 달리, 한국어는 조사, 어미, 복합어 등의 형태적 다양성이 커서 같은 양의 데이터로는 학습이 어려웠다.

**2. BPE 학습 방식의 한계**

- 현재 구현한 BPE는 단순한 빈도 기반 병합 방식으로, 한국어의 형태소적 특성을 충분히 반영하지 못했다.
- 결과적으로 <unk> 토큰이 여전히 다수 발생하고, 의미 단위가 제대로 분리되지 않는 현상이 지속되었다.

**3. 모델 구조 및 학습 방식의 한계**

- TransformerLanguageModel의 구조와 학습 루프가 영어 중심으로 설계되어 있었고, 한국어 데이터에 최적화되어 있지 않았다.
- 데이터 양이 증가해도 모델이 한국어의 문법적 패턴을 효과적으로 학습하지 못하는 모습이 관찰되었다.

**4. 과적합과 일반화 실패**

- Loss는 지속적으로 감소했으나, 실제 생성 결과에서는 반복(repetition) 현상이 두드러지고 문법적으로 올바른 문장이 거의 생성되지 않았다. 이는 모델이 학습 데이터를 외우는 수준에 머무른 것으로 판단된다.

### 결정 및 대응

3단계에서 한국어 BPE를 충분히 고도화하는 데 예상보다 많은 시간과 노력이 소요될 것으로 판단했다.

이에 따라 **3단계는 최소한으로 정리**하고 4단계로 넘어가기로 결정했다. 구체적으로는:

- 현재 283문장 코퍼스와 Tokenizer 학습 코드는 유지
- 모델 학습 및 생성 확인 부분은 주석 처리하거나 최소화
- 3단계의 시도와 한계를 트러블 슈팅으로 기록하여 추후 참고할 수 있도록 남김

### 인사이트

- 한국어를 from-scratch로 다루는 것은 영어보다 훨씬 더 많은 데이터와 세심한 전처리가 필요하다는 것을 직접 체감했다.
- 단순히 데이터 양을 늘리는 것만으로는 BPE의 품질이 크게 개선되지 않을 수 있다. 한국어의 형태적 특성을 고려한 전처리나, 이미 검증된 한국어 BPE 도구를 활용하는 방향도 고려할 필요가 있다.
- 제한된 시간 내에 모든 단계를 완벽하게 수행하기보다는, **각 단계의 핵심 목표를 명확히 하고, 달성하기 어려운 부분은 과감하게 정리**하는 판단도 중요하다는 것을 배웠다.
- 3단계에서 기대한 만큼의 결과를 내지 못했지만, 이 과정에서 한국어 자연어 처리의 난이도를 현실적으로 인식하게 된 것은 이후 학습 방향을 설정하는 데 도움이 될 것으로 기대한다.

## 트러블 슈팅 17 - 한국어 데이터로 모델 학습 시도 및 과적합 현상

### 문제 상황

3단계에서 BPE Tokenizer를 한국어 데이터로 학습한 후, 4단계에서 해당 Tokenizer를 사용하여 TransformerLanguageModel을 한국어 데이터로 학습해보았다.

목표는 한국어 문장을 입력받았을 때 그럴듯한 한국어 문장을 생성하는 것이었으나, 학습 결과 Loss는 크게 감소했음에도 불구하고 실제 생성 품질은 기대에 미치지 못했다.

### 결정 및 대응

**1. 한국어 데이터로 모델 학습 진행**

기존에 학습된 한국어 BPE Tokenizer(vocab_size=488)를 사용하여 283문장의 한국어 데이터로 모델을 50 epoch 학습했다.

학습 결과는 다음과 같았다:

- **Loss**: 4.84 → **0.067** (매우 낮게 수렴)
- **생성 결과**:
  - 안녕 → <unk><unk> <unk> 제<unk>로 사면서 기존 폰은 동생에게
  - 오늘 날씨가 → 오늘 날씨가 했다. 같아. 으로 먹은 파스타는 으로 먹
  - 나는 좋아해 → 나는 좋아해 배우고 있다. 여름휴가방식 재미있어서 오늘은 일찍 집

Loss는 크게 떨어졌으나, 생성된 문장은 여전히 <unk>가 다수 포함되어 있고, 문맥이 맞지 않으며, 학습 데이터의 조각이 무작위로 조합되는 현상이 나타났다.

**2. 결과 분석 및 4단계 정리 결정**

Loss가 매우 낮음에도 불구하고 생성 품질이 좋지 않은 주요 원인은 다음과 같다고 판단했다:

- **BPE Tokenizer의 vocabulary 품질 부족**: 한국어 데이터로 학습했으나, 여전히 <unk> 발생률이 높고 의미 있는 subword 단위로 분해되지 않음.
- **데이터 규모의 한계**: 283문장으로는 한국어의 문법적·의미적 다양성을 충분히 학습하기 어려움.
- **과적합(Overfitting) 현상**: 모델이 학습 데이터를 거의 외워버린 상태로, 일반화 능력이 크게 떨어짐.

이에 따라 **4단계도 최소 정리 수준으로 마무리**하고 5단계로 넘어가기로 결정했다.

### 인사이트

- Loss가 낮다고 해서 반드시 생성 품질이 좋은 것은 아니라는 점을 다시 확인했다. 특히 작은 데이터셋에서는 **과적합**이 쉽게 발생하며, 이는 생성 단계에서 더 큰 문제를 일으킨다.
- 한국어를 from-scratch로 다루는 경우, BPE Tokenizer의 품질이 모델 학습 결과에 매우 큰 영향을 미친다는 것을 체감했다. Tokenizer 품질이 낮으면 모델이 학습할 수 있는 정보 자체가 제한된다.
- 현재 구조(모델 아키텍처 + 학습 방식 + 데이터 규모)로는 한국어 생성 품질을 의미 있게 끌어올리기 어렵다는 현실적인 한계를 인식하게 되었다.
- 제한된 시간 내에 모든 단계를 고도화하는 것보다, **각 단계에서 달성 가능한 수준을 명확히 하고 정리하는 판단**이 중요하다는 점을 다시 한 번 확인했다.

## 트러블 슈팅 18 - 사전학습된 Transformer에 Instruction Tuning 적용

### 문제 상황

- 기존 `korean_model.pt`는 `korean_corpus.txt`(독립된 일상 대화 문장 283개)를 next-token prediction 방식으로 학습한 순수 사전학습(pre-training) 모델이었다.
- 이 모델을 RAG_Project의 generator 자리에 끼우려 했으나, "질문에 답하라"는 프롬프트 구조 자체를 모델이 지시(instruction)로 인식하지 못하고, 그냥 자연스럽게 이어지는 문장을 생성하려는 경향을 보일 것으로 예상되었다.
- 즉, 모델이 "지시-응답" 패턴을 학습한 적이 없어, RAG 파이프라인에 연결해도 의미 있는 응답을 기대하기 어려운 상태였다.

### 원인 분석

1. **데이터 구조의 한계**
   - `korean_corpus.txt`는 질문-답변 쌍이 아니라 독립된 평서문/의문문이 나열된 형태였다.
   - `train.py`의 학습 루프(`inputs = input_ids[:, :-1]`, `labels = input_ids[:, 1:]`)는 전체 시퀀스에 대해 loss를 계산하는 순수 next-token prediction 방식이라, "질문 뒤에 답변이 와야 한다"는 역할 구분 자체가 학습 신호에 존재하지 않았다.
2. **vocab 호환성 제약**
   - 처음에는 기존 `korean_model.pt`의 가중치를 보존하면서 instruction 데이터를 추가하려 했으나, `BPETokenizer.train()`이 vocab을 corpus로부터 매번 새로 계산하는 구조라, 새 구조 토큰(`[질문]:` 등)을 추가하면 vocab_size가 바뀌어 `nn.Embedding(vocab_size, d_model)`의 shape가 기존 저장된 가중치와 불일치하게 되는 문제가 있었다.
   - → 기존 가중치를 보존하지 않기로 결정하면서 이 제약은 해소되었다.
3. **소규모 데이터에서의 분산(variance) 문제**
   - 일상 대화 전반을 다루는 넓은 도메인으로 QA 쌍을 만들면, 20~30개 수준의 적은 데이터로는 질문 구조와 답변 구조가 매번 달라져 패턴 학습이 어려울 것으로 판단되었다.

### 결정 및 대응

1. **vocab 제약 해제**
   - 기존 `korean_model.pt`를 보존하지 않고, tokenizer와 모델 모두 새 데이터로 처음부터 재학습하기로 결정.
2. **도메인을 "일정 묻기" 템플릿 하나로 좁힘**
   - 질문 템플릿을 `오늘/내일 [활동] 할 거야?` 하나로 고정하고, 답변은 수락(`응, ~`)/거절(`아니, ~ 대신 ~`) 두 패턴으로 한정.
   - 활동 어휘는 기존 `korean_corpus.txt`에 등장한 단어(산책, 운동, 조깅, 쇼핑 등)를 재사용.
   - 수락:거절 비율을 17:7로 비대칭 구성 (단순 반복이 아니라 두 가지 응답 분기를 모델이 구분하는지 보기 위함).
3. **Loss Masking 구현 (`build_qa_training_pair()`)**
   - 질문+답변을 이어붙인 시퀀스에서, **답변 영역에만 loss를 계산**하도록 마스킹 로직을 별도 함수로 분리해 구현.
   - 핵심 코드:

     ```python
     full_ids = question_ids + answer_ids
     inputs = full_ids[:-1]
     labels = full_ids[1:]

     mask_boundary = len(question_ids) - 1
     masked_labels = [
        label if idx >= mask_boundary else IGNORE_INDEX
        for idx, label in enumerate(labels)
     ]
     ```

- `nn.CrossEntropyLoss(ignore_index=-100)`를 사용해, 마스킹된 질문 영역이 loss 계산에서 자동으로 제외되도록 함.
- 마스킹 경계(`mask_boundary`)와 관련된 7개 단위 테스트(`test_qa_collate.py`)로 정상 동작을 사전에 검증.

1. **학습 실행 및 결과 확인**
   - 24쌍, 50 epoch 학습 결과, loss가 2.8929(1 epoch) → 0.0032(50 epoch)로 감소.
   - 생성 결과 예시:

     ```bash
     Prompt: 오늘 산책 할 거야?
     Generated: 오늘 산책 할 거야? 아니, 조깅 대신 산책 할 거야 할 거야 할 거야...
     ```

     - **성공한 부분**: 모델이 질문(`?`) 뒤에 `응,` / `아니,`로 시작하는 답변 구조를 생성하기 시작함. 이는 사전학습 단계에서는 없었던, instruction tuning을 통해 새로 학습된 패턴.
     - **한계 1 (내용 정확도)**: "오늘 산책"의 정답은 `응, 산책 할 거야`이나, 어휘를 공유하는 다른 쌍(`내일 조깅` → `아니, 조깅 대신 산책 할 거야`)과 혼동되어 틀린 내용을 생성. vocab_size=55, 24쌍이라는 극소 규모에서 발생한 과적합(overfitting)으로 판단. (한계점)
     - **한계 2 (반복 생성)**: `<eos>` 토큰이 없어 `generate()`가 멈출 신호를 갖지 못해, `max_new_tokens`까지 같은 단어를 반복 생성. (한계점, 후속 트러블슈팅에서 해결 진행 중)

### 인사이트

- Instruction tuning은 모델 구조를 바꾸는 게 아니라 **데이터 형식(질문-답변 쌍)과 loss 계산 범위(답변 영역만)를 바꾸는 fine-tuning의 한 형태**라는 점을 직접 구현하며 확인했다.
- 데이터 규모가 극히 작을 때는, 다양한 주제를 다루기보다 **질문 템플릿 하나로 좁혀서 슬롯만 바꾸는 방식**이 패턴 학습에 훨씬 유리하다는 것을 결과로 확인했다 (loss가 빠르게 0에 가까워짐).
- 다만 loss가 매우 낮아졌다는 것이 "모델이 잘 작동한다"를 의미하지 않는다는 점도 함께 확인했다 — 오히려 데이터가 작을 때는 loss 급감이 과적합의 신호일 수 있으며, 실제 생성 결과에서 어휘 공유로 인한 답변 혼동이 그대로 드러났다.
- "한계점(개선 가능)"과 "주의사항(구조적 한계)"을 구분해서 기록하는 것이, 다음에 어디를 고쳐야 하는지(예: EOS 추가는 가능한 개선, 24쌍 규모의 근본적 데이터 부족은 당장 해결하지 않기로 한 범위 밖 문제)를 명확히 하는 데 도움이 되었다.
- 핵심 로직(loss masking)을 모델 학습 루프에 바로 넣지 않고 **독립된 순수 함수(`build_qa_training_pair`)로 분리해 먼저 단위 테스트로 검증**한 뒤 통합한 것이, 실제 학습 실행 전에 경계값 오류(off-by-one)를 사전에 잡는 데 효과적이었다.

## 트러블 슈팅 19 - `generate()` 반복 생성 문제 해결 (EOS 토큰 도입)

### 문제 상황

- 이전 트러블슈팅(Instruction Tuning 적용)에서 학습한 모델이, 답변이 끝나야 할 지점 이후에도 계속 같은 단어(`할 거야`)를 반복 생성하는 문제가 있었다.
- 예: `Prompt: 오늘 산책 할 거야?` → `Generated: ... 아니, 조깅 대신 산책 할 거야 할 거야 할 거야 할 거야 할 거야 할`
- `TransformerLanguageModel.generate()`에는 애초에 "여기서 멈춰라"는 신호를 처리하는 로직 자체가 없었고, 이는 `generate()` docstring에 이미 known limitation으로 명시되어 있었다 — `EOS 토큰 기반 early stopping이 구현되어 있지 않음. 따라서 max_new_tokens만큼 무조건 생성함.`

### 원인 분석

1. **vocabulary에 종료 신호가 없음**
   - `BPETokenizer`는 `<unk>` 외에 다른 특수 토큰을 예약하지 않았다. 따라서 모델이 학습 과정에서 "답변 다음에 무엇이 와야 하는가"를 배울 대상 자체가 없었다.
2. **학습 데이터에 종료 지점이 표시되어 있지 않음**
   - `build_qa_training_pair()`로 답변 영역에 loss를 계산하긴 했지만, 답변 토큰 시퀀스 자체에 "끝"을 의미하는 토큰이 포함되어 있지 않아, 모델은 답변이 어디서 끝나야 하는지에 대한 학습 신호를 받지 못했다.
3. **`generate()` 루프에 멈춤 조건이 없음**
   - 토큰을 생성하는 `for` 루프가 `max_new_tokens`로만 종료 조건을 가지고 있어서, 설령 모델이 종료를 의미하는 토큰을 생성하더라도 그걸 해석해서 멈추는 코드가 없었다.

### 결정 및 대응

1. **Tokenizer에 `<eos>` 토큰 예약**
   - `BPETokenizer.__init__`에 `eos_token="<eos>"` 파라미터를 추가하고, `train()`에서 `<unk>=0`, `<eos>=1`로 고정 예약한 뒤, 나머지 서브워드는 ID 2부터 부여하도록 변경.
   - **개선한 코드**:

     ```python
     # 수정 전
     self.token_to_id = {self.unk_token: 0}
     self.id_to_token = {0: self.unk_token}
     ...
     for idx, token in enumerate(sorted_tokens, start=1):

     # 수정 후
     self.token_to_id = {self.unk_token: 0, self.eos_token: 1}
     self.id_to_token = {0: self.unk_token, 1: self.eos_token}
     ...
     for idx, token in enumerate(sorted_tokens, start=2):
     ```

2. **학습 데이터의 답변 끝에 `<eos>` 추가**
   - `train.py`에서 답변을 인코딩한 뒤, 끝에 `eos_id`를 붙여서 `build_qa_training_pair()`에 전달하도록 수정.
   - **개선한 코드**:

     ```python
     raw_answer_ids = tokenizer.encode(answer)
     if len(question_ids) == 0 or len(raw_answer_ids) == 0:
        num_skipped += 1
        continue
     answer_ids = raw_answer_ids + [eos_id]
     ```

     - 이렇게 하면 답변 영역의 마지막 위치에서 모델이 "다음 토큰은 `<eos>`다"를 예측하도록 학습되고, 이 위치는 이미 답변 영역에 속하므로 기존 loss masking 로직(질문 영역만 제외)을 그대로 재사용할 수 있었다.

3. **`generate()`에 조기 종료(early stopping) 로직 추가**
   - `eos_token_id` 파라미터를 추가하고, 매 스텝마다 생성된 토큰이 `eos_token_id`와 일치하면 즉시 `break`하도록 수정.
   - **개선한 코드**:

     ```python
     input_ids = torch.cat([input_ids, next_token], dim=1)

     if eos_token_id is not None and next_token.item() == eos_token_id:
        break
     ```

     - `eos_token_id=None`일 경우 기존 동작(무조건 `max_new_tokens`만큼 생성)을 그대로 유지하도록 해서, 하위 호환성(backward compatibility)을 깨지 않도록 했다.

4. **종료 처리 로직을 독립된 순수 함수로 분리**
   - "생성된 시퀀스에서 eos 이후를 잘라낸다"는 로직을 `generate()` 내부가 아니라 `trim_after_eos()`라는 별도 함수로 분리했다.
   - 분리한 이유: `generate()` 전체를 테스트하려면 학습된 모델 가중치가 필요해 단위 테스트가 어렵지만, "eos 이후를 자른다"는 규칙 자체는 모델과 무관한 순수 로직이라 모델 없이도 경계 케이스(맨 앞에 eos, 중간에 eos, eos 없음, eos 여러 번 등장)를 테스트할 수 있었다.
5. **단위 테스트로 사전 검증**
   - `trim_after_eos()`에 대한 5개 테스트로 경계 케이스(중간/맨 앞/없음/다중 등장/빈 입력)를 모두 검증.
   - `generate()`의 조기 종료 자체는 `torch.multinomial` 기반 확률적 샘플링이라 보장된 재현이 어려워, `vocab_size`를 의도적으로 작게 줄이고 시드를 고정해 `eos`가 반드시 등장하는 케이스를 별도로 구성해(`테스트 4-1`) "eos 등장 시 즉시 멈춘다"를 결정적으로(deterministic) 검증했다.
6. **실제 24쌍 데이터로 재학습하여 최종 확인**
   - vocab_size 56 (기존 55 + `<eos>` 1개 추가), 50 epoch 학습 결과 loss가 2.3319(1 epoch) → 0.0029(50 epoch)로 감소.
   - 생성 결과:

     ```python
     Prompt: 오늘 산책 할 거야?
     Generated: 오늘 산책 할 거야? 아니, 헬스장 대신 산책 할 거야

     Prompt: 내일 운동 할 거야?
     Generated: 내일 운동 할 거야? 아니, 운동 할 거야

     Prompt: 오늘 회의 할 거야?
     Generated: 오늘 회의 할 거야? 응, 회의 할 거야
     ```

   - **한계점 2(반복 생성) 해결 확인**: 세 프롬프트 모두 답변이 끝난 지점에서 정확히 생성이 멈췄다. 이전처럼 `할 거야`가 무한 반복되는 현상이 사라짐 — `<eos>` 토큰 도입과 `generate()`의 조기 종료 로직이 의도대로 동작함을 실제 학습 결과로 확인했다.
   - **한계점 1(내용 정확도)은 일부 개선, 일부 잔존**: `오늘 회의` → `응, 회의 할 거야`는 데이터와 정확히 일치. 반면 `내일 운동` → `아니, 운동 할 거야`는 "아니"로 시작하면서도 같은 활동(운동)을 다시 언급하는, 거절 답변 형식과 수락 답변 내용이 섞인 형태로 나타났다. 이는 이전 트러블슈팅에서 진단한 "24쌍 규모에서의 어휘·패턴 혼동"이 여전히 일부 남아 있음을 보여준다. (주의 사항 — 데이터 규모의 구조적 한계이며, 이번 트러블슈팅의 해결 범위 밖)

### 인사이트

- 모델이 "언제 멈춰야 하는지"를 알게 하려면, 멈춤 신호가 **vocabulary(토큰으로서 존재) → 학습 데이터(정답으로 한 번이라도 등장) → 생성 로직(만나면 해석해서 멈춤)** 이렇게 세 단계 모두에 일관되게 반영되어야 한다는 것을 직접 구현하며 확인했다. 한 단계만 빠져도(예: tokenizer에는 있지만 학습 데이터 끝에 안 붙이면) 동작하지 않는다.
- 확률적 샘플링이 들어간 코드(`generate()`)를 테스트할 때는, "항상 일어나는 일"을 검증하기보다 "일어났을 때 올바르게 처리되는가"와 "결정적으로 재현 가능한 최소 설정에서는 반드시 일어나는가"를 나눠서 검증하는 게 효과적이었다. 단순히 시드만 고정해서는 충분하지 않았고, vocab_size 같은 환경 변수도 같이 조정해야 재현성이 확보되었다.
- 단위 테스트로 사전에 검증한 로직(`<eos>` 예약, `trim_after_eos`, `generate()`의 break 조건)이 실제 학습·생성 결과에서도 동일하게 작동함을 확인하면서, **"작은 단위에서 옳다는 것"과 "전체 파이프라인에서 옳게 동작한다는 것" 사이의 간극이 이번 경우엔 크지 않았다**는 것을 직접 경험했다. 이는 각 단위를 독립적으로 테스트 가능하게 설계해둔 효과로 볼 수 있다.
- "반복 생성"과 "내용 정확도"는 서로 다른 원인(생성 종료 시점 vs. 데이터 규모로 인한 과적합)에서 비롯된, 분리된 문제라는 것이 이번 결과로 명확해졌다. 하나(반복 생성)를 고쳤다고 다른 하나(내용 정확도)가 같이 해결되지 않는다는 점에서, 한계점을 기록할 때 "같은 증상처럼 보여도 원인이 다르면 별개의 항목으로 추적해야 한다"는 점을 다시 확인했다.
