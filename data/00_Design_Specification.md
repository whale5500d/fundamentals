# Virtual document design specification

## (txt/md) DaySync Manual([살펴보기](./daysync_manual.md))

| 속성      | 값                                                                                                                                   |
| --------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| 문서 유형 | 가상 일정 관리 시스템 매뉴얼                                                                                                         |
| 형식      | Markdown (.md)                                                                                                                       |
| 구성      | 시스템 개요, 동작 방식, API 사용법, 설정 항목, 일정 충돌 처리                                                                        |
| 목적      | 모델(Gemma 4 또는 커스텀 Transformer)이 절대 모를 가상의 고유 정보(코드네임, 설정값, 충돌 코드 등)를 포함시켜 RAG 효과를 명확히 검증 |

(가상 문서 설계 핵심) 이 매뉴얼에는 모델이 절대 알 수 없는 고유 정보들이 의도적으로 박혀있어, RAG가 제대로 동작하는지 여부를 비교적 명확하게 확인할 수 있음.

**RAG 검증용 핵심 정보(Verification Anchors)**

| 정보                                   | 값                                                | 검증 시 사용                                         |
| -------------------------------------- | ------------------------------------------------- | ---------------------------------------------------- |
| 시스템 내부 코드네임                   | "프로젝트 새벽별(Project Dawnstar)"               | RAG 없이 물으면 "모른다"고 답해야 정상               |
| `preference_threshold` 기본값          | `0.65`                                            | 문서 없이는 추측 불가능한 고유 설정값                |
| 추천 주기(recommendation_cycle) 기본값 | 7일                                               | 구체적인 수치 — 환각(hallucination) 여부 판별에 좋음 |
| 기본 API 포트                          | 9221                                              | 일반적인 포트(80, 443, 8080)가 아니므로 추측 불가    |
| 충돌 코드 SC-114 의미                  | "동일 요일에 거절/수락 응답이 동시에 기록된 경우" | 코드와 의미를 정확히 매칭하는지 검증                 |
| 일정 충돌 시 채택 규칙                 | "가장 마지막에 들어온 응답을 최종 값으로 채택"    | 완전히 가상의 운영 규칙                              |

이 문서를 통해 다음과 같은 질문으로 RAG 효과를 검증할 수 있음.

- "DaySync의 내부 코드네임은 무엇인가?" (RAG는 모르는 정보)
- "충돌 코드 SC-114는 무엇을 의미하는가?" (RAG 없이는 지어낼 가능성)
- "preference_threshold의 기본값은 무엇인가?" (구체적 수치 - 환각 여부 명확히 드러남)

## (txt/md) DaySync Team Records([살펴보기](./daysync_team_records.md))

| 속성      | 값                                                                               |
| --------- | -------------------------------------------------------------------------------- |
| 문서 유형 | 가상 팀별 일정 운영 기록                                                         |
| 형식      | Markdown (.md)                                                                   |
| 구성      | 팀별 운영 현황(3개 팀), 충돌 코드 통계, Summary Table(Graph RAG 추출 대상)       |
| 목적      | Graph RAG(Step C-4)의 entity/relation 추출 및 그래프 탐색(키워드 매칭 기반) 검증 |

(가상 문서 설계 핵심) 동명이인(Team Sunrise의 "도윤"과 Team Lumen 매니저 "도윤")을 의도적으로 배치하여, 섹션 경계를 넘어 서로 다른 인물의 정보가 섞이지 않는지 검증할 수 있도록 설계함 (기존 NimbusFlow 문서의 "8842/Drift Score 섹션 분리" 검증과 동일한 목적).

**Graph RAG 검증용 관계(Relations)**

| Entity A      | Relation Type        | Entity B                          | 검증 시 사용                                                   |
| ------------- | -------------------- | --------------------------------- | -------------------------------------------------------------- |
| 도윤(Sunrise) | prefers_activity     | 산책                              | 선호 활동 추출 정확도                                          |
| 도윤(Sunrise) | managed_by           | 서연                              | 관리자 관계 추출 정확도                                        |
| 도윤(Sunrise) | experienced_conflict | SC-114                            | 멀티홉(multi-hop) 탐색 검증용 (도윤 -> 충돌 코드 -> 충돌 의미) |
| 민준          | changed_config       | preference_threshold: 0.65 -> 0.5 | 설정 변경 이력 추출 정확도                                     |

이 문서를 통해 다음과 같은 질문으로 Graph RAG 효과를 검증할 수 있음.

- "Team Sunrise를 관리하는 사람은 누구인가?" (1-hop 탐색)
- "SC-114를 겪은 팀원의 관리자는 누구인가?" (2-hop 탐색 — 도윤 -> SC-114, 도윤 -> 서연)
- "Team Lumen의 매니저 이름은?" (동명이인 함정 — Team Sunrise의 도윤과 혼동하지 않아야 함)
