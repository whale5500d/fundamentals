## 트러블 슈팅 1 - 로컬 시크릿 스캔 도구 부재

### 문제 상황

`gitleaks`, `git filter-repo` 둘 다 기본 설치되어 있지 않아 명령 실행 시 `command not found` 발생.

### 고려한 옵션

- Homebrew로 설치
- 바이너리 직접 다운로드 (gitleaks)
- pip로 설치 (git-filter-repo)

### 결정 및 이유

Homebrew를 우선 시도하고, 안 될 경우의 대안(바이너리 직접 다운로드, pip)을 함께 준비함. 두 도구 모두 저장소 히스토리 재작성·시크릿 탐지에 전용화된 도구라 수동 검토보다 정확도가 높아 도입을 결정함.

## 트러블 슈팅 2 - 원본 저장소 보존을 위한 임시 작업공간 전략

### 문제 상황

`git filter-repo`가 파괴적(destructive) 도구라, 원본 `00_BASIC`에서 직접 실행하면 지정하지 않은 나머지 폴더(algorithm, math 등)가 히스토리에서 영구 소실되는 문제가 있었음.

### 고려한 옵션

- 원본에서 직접 filter-repo 실행 후 필요할 때마다 git으로 복구 시도
- 원본은 그대로 두고, 매 작업 단위마다 새 clone(임시 사본)에서만 filter-repo 실행

### 결정 및 이유

매번 새 clone 방식을 채택함. 원본을 "여러 번 복사해서 꺼내 쓸 수 있는 무결한 데이터 소스"로 유지하고, 파괴적 재작성은 일회용 임시 사본에서만 발생하도록 격리하는 것이 안전함. clone 반복에 따른 디스크·시간 비용은 원본 손상 리스크에 비해 감수할 만하다고 판단함.

## 트러블 슈팅 3 - main 브랜치 직접 작업의 위험성과 전략 전환

### 문제 상황

초기에 `fundamentals` 저장소의 `main` 브랜치에서 직접 병합 작업을 진행하려 했으나, 실험적 성격의 작업(병합 결과 검증 전)을 안전판 없이 진행하는 데 대한 우려가 제기됨.

### 원인 분석

`main`에서 직접 `git subtree`/`merge`를 실행하면, 결과가 잘못됐을 때 되돌리는 과정에서 실수 위험이 커짐. 이후 작업 단위(TIL, algorithm, math)별로 브랜치를 나눌지, 전체를 포괄하는 브랜치 하나로 진행할지에 대한 판단도 추가로 필요해짐.

### 결정 및 대응

1차로 단위별 브랜치(`merge-basic`)를 시도했으나, 개인 저장소·단독 작업이라는 특성상 브랜치 생성·정리 반복이 불필요한 오버헤드로 판단되어, 이후 전체를 포괄하는 단일 브랜치(`consolidate-fundamentals`)로 전환함. 대신 커밋은 작업 단위별로 분리해 추적 가능성을 유지함.

### 인사이트

브랜치 전략은 협업 규모와 리뷰 필요성에 따라 달라진다. 리뷰어가 없는 개인 저장소에서는 브랜치 세분화보다 커밋 단위 분리가 더 실용적인 안전장치가 될 수 있다.

## 트러블 슈팅 4 - 병합 대상(TIL) 스코프 오인

### 문제 상황

`git subtree add --prefix=TIL ...` 작업을 `00_BASIC`의 일부를 옮기는 작업으로 오인한 채 진행하려 했음.

### 원인 분석

애초 전체 폴더 구조(`01_dev_project` 등)에 TIL이라는 이름의 저장소가 명시적으로 없었는데, 목표 구조(`fundamentals`)에는 `TIL`이 최상위 항목으로 존재해 소스가 불명확한 상태에서 작업이 진행됨. 확인 결과 TIL은 `00_BASIC`과 무관한 별도 저장소이며, 내용 자체도 재정의(부트캠프 학습 내용 등)가 필요한 상태였음.

### 결정 및 대응

TIL 병합을 보류하고, `00_BASIC` 관련 작업(algorithm, math)을 먼저 진행하는 순서로 조정함.

### 인사이트

목표 구조와 현재 소스 구조 사이의 매핑이 명확하지 않은 상태에서 병합 명령을 먼저 실행하면, 되돌리는 비용이 더 크다. 매핑표를 먼저 확정하는 단계가 실제 명령 실행보다 선행되어야 함.

## 트러블 슈팅 5 - 경로 매핑 오류로 인한 재작업

### 문제 상황

`04_algorithm`을 `06_Programing_Language/03_Algorithms/algorithm`으로 rename했으나, 실제로는 `03_Algorithms` 바로 아래 하위 폴더(`01_선형_자료구조` 등)가 위치해야 했음. `algorithm`이라는 불필요한 경로 한 겹이 추가로 들어감.

### 원인 분석

python/java/javascript처럼 형제 폴더가 여러 개인 경우엔 구분용 폴더명이 필요했지만, `04_algorithm`은 소스가 하나뿐이라 목적지 경로 자체가 `03_Algorithms`여야 했음. 앞선 패턴을 그대로 답습하며 목적지 경로에 불필요한 접미사를 추가한 것이 원인.

### 결정 및 대응

`git reset --hard HEAD~1`로 잘못된 merge 커밋을 되돌리고, 임시 사본을 폐기 후 경로를 수정해(`--path-rename 04_algorithm:06_Programing_Language/03_Algorithms`) 재작업함.

### 인사이트

경로 rename 작업은 소스 폴더 하나당 목적지 폴더가 몇 개인지(1:N인지 1:1인지)에 따라 규칙이 달라진다. 반복 작업 시 이전 패턴을 기계적으로 답습하지 않고, 매번 목적지 구조를 사전에 `ls`로 확인하는 절차가 필요함.

## 트러블 슈팅 6 - 작업 순서 실수로 인한 커밋 롤백

### 문제 상황

CHANGELOG에 algorithm 작업 기록을 남기기 전에 math 병합(`git merge`)을 먼저 진행함. 작업 기록 순서가 실제 진행 순서와 어긋남.

### 원인 분석

CHANGELOG 갱신을 "작업 완료 후 별도로 몰아서 하는 것"으로 여기고 진행하다, 단위별로 즉시 기록해야 한다는 원칙이 뒤늦게 확인됨.

### 결정 및 대응

`git reset --hard HEAD~1`로 math merge 커밋을 되돌리고, algorithm 작업에 대한 CHANGELOG 커밋을 먼저 넣은 뒤 math 작업을 재개함.

### 인사이트

"작업 완료 → 결과 확인 → CHANGELOG 커밋"을 하나의 반복 단위로 고정하면, 순서가 어긋나는 문제 자체가 재발하지 않는다.

## 트러블 슈팅 7 - gitleaks report 파일 위치·경로 혼동

### 문제 상황

`-r gitleaks-report.json`(상대경로)으로 실행 후 다른 위치에서 `cat`을 시도해 파일을 못 찾음. 또한 `--source /path/to/repo`를 예시 그대로 실행해 "no such file or directory" 에러 발생.

### 원인 분석

`-r`의 출력 경로가 스캔 대상(`--source`)과 무관하게 명령 실행 시점의 현재 위치(cwd) 기준으로 결정된다는 점을 인지하지 못함. 두 번째 사례는 안내 예시의 placeholder를 실제 경로로 치환하지 않고 그대로 복사·실행한 것이 원인.

### 결정 및 대응

이후 모든 gitleaks 실행 시 `-r`에 절대경로(`~/gitleaks-*.json`)를 고정 지정하도록 규칙화함.

### 인사이트

반복 실행하는 명령의 출력 경로는 상대경로 대신 절대경로로 고정하는 것이, 작업 위치가 자주 바뀌는 멀티 저장소 작업 환경에서 사소한 실수를 구조적으로 방지하는 방법이 됨.

## 트러블 슈팅 8 - 작업 성격을 표현하는 용어 정립

### 문제 상황

여러 저장소를 하나로 합치는 작업을 계속 "마이그레이션"이라고 통칭하고 있었으나, 계정 간 1:1 이전(표 33)과 성격이 다름을 인지함.

### 고려한 옵션

- 계속 "마이그레이션"으로 통칭
- 별도 용어로 구분 (repository consolidation / monorepo consolidation)

### 결정 및 이유

"Repository consolidation"이라는 용어를 채택함. 1:1 이전과 N:1 병합을 구분해서 기록해야, 이후 CHANGELOG·포트폴리오 문서에서 실제로 수행한 작업의 성격이 정확히 전달된다고 판단함.

## 트러블 슈팅 9 - 이관 완료된 원본 저장소의 처리 방침

### 문제 상황

`00_BASIC`의 필요한 내용이 전부 `fundamentals`로 흡수된 후, 원본 저장소를 어떻게 처리할지 결정이 필요했음.

### 고려한 옵션

- 완전 삭제
- 그대로 방치
- GitHub Archive(읽기 전용 전환)

### 결정 및 이유

Archive를 선택함. 삭제 시 원본 커밋 히스토리라는 참조 증거가 영구 소실되는 반면, Archive는 언제든 Unarchive로 복구 가능한 가역적(reversible) 조치라 리스크가 가장 낮다고 판단함. 로컬 사본은 전체 이관 작업이 끝난 뒤 별도로 정리하기로 함.

## 트러블 슈팅 10 - 목표 구조와 실제 구조 간 경로 명칭 불일치

### 문제 상황

`00_BASIC`의 python/java/javascript를 옮기며 목적지 경로를 `Programming_Language/basic/python` 등으로 지정해 `filter-repo`를 실행했으나, 실제 `fundamentals` 저장소의 폴더명은 `06_Programing_Language/01_Basic`(번호 접두사 포함, `Programming`이 아닌 `Programing` 스펠링)이었음.

### 원인 분석

목적지 구조를 실제로 확인하지 않고, 이전 대화에서 제시된 목표 구조 설명(`Programming_Language`)을 그대로 명령어에 옮겨 적은 것이 원인. `fundamentals` 저장소의 실제 폴더 트리를 사후에 공유받고 나서야 번호 접두사와 오타(`Programing`)가 실제 폴더명에 고정되어 있다는 걸 확인함.

### 결정 및 대응

`--path-rename`의 목적지 경로를 실제 폴더명(`06_Programing_Language/01_Basic/...`)에 맞춰 재작성함. 이미 실행한 `filter-repo`는 임시 clone 내부 작업이라 폐기 후 재clone으로 원상복구, 원본 손상 없이 재작업함.

### 인사이트

`--path-rename`처럼 문자열을 정확히 일치시켜야 하는 명령은, 실행 전에 반드시 목적지 저장소의 실제 `ls -R` 결과로 폴더명을 대조해야 한다. 사람이 부르는 이름(구두 설명)과 실제 파일시스템의 이름이 다를 수 있다는 전제를 깔고 진행해야 함.

## 트러블 슈팅 11 - CHANGELOG 문서 구조 설계

### 문제 상황

`00_Meta/CHANGELOG.md`에 작업 이력을 남기려 했으나, 초기 형식(날짜를 헤더로 사용)에서 같은 날짜에 여러 작업이 이어질 때 `## 2026-07-15 (계속)`처럼 헤더가 중복 생성되는 문제, 그리고 "다음 예정" 항목이 완료될 때마다 과거 기록을 다시 찾아 수정해야 하는 문제가 발생함.

### 고려한 옵션

- 날짜 헤더 유지 + "진행 상황" 섹션을 별도로 분리해 체크박스로 관리하는 방식
- 날짜 옆에 작업 성격(예: Repository Consolidation)을 괄호로 표기하고, 그 아래 단위별 항목을 리스트로만 나열하는 단순한 방식

### 결정 및 이유

후자를 선택함. 별도 "진행 상황" 섹션 분리안은 구조적으로는 더 견고하지만 지금 규모(개인 저장소, 단독 작업)에 비해 과한 설계로 판단됨. 날짜 헤더를 작업 단위로 재사용하고, 각 항목 아래 한 단계 들여쓰기로 변경/결정 사항을 구분하는 최소 구성으로 확정함.

## 트러블 슈팅 12 - 홈 디렉토리 절대경로(`~`) 개념

### 배경

`git clone` 목적지를 `~/00_BASIC_temp`로 지정했는데, 이게 정확히 어느 위치에 생성되는지 불명확해 확인이 필요했음.

### 개념 설명

`~`(틸드, tilde)는 현재 로그인한 사용자의 홈 디렉토리(home directory, 맥 기준 보통 `/Users/사용자이름`)를 가리키는 셸(shell) 축약 표기다. 이는 절대경로(absolute path)이므로, 터미널이 현재 어느 폴더에 위치해 있든 상관없이 항상 동일한 위치를 가리킨다. 반대로 `00_BASIC_temp`처럼 접두사 없이 쓰면 상대경로(relative path)가 되어, 현재 위치에 따라 실제 생성 위치가 달라진다.

### 적용

이후 반복 작업에서 경로 혼동을 줄이기 위해, temp 폴더 생성이나 report 파일 출력처럼 위치가 중요한 명령에는 `~/` 절대경로를 의도적으로 사용하는 원칙을 세움.

## 트러블 슈팅 13 - `cp -r ... $(date +%Y%m%d)` 명령어 분해

### 배경

`fundamentals` 저장소를 `filter-repo`로 재작성하기 전에, 안전을 위한 백업본을 만드는 명령어의 각 구성요소를 이해할 필요가 있었음.

### 개념 설명

`cp -r . ../fundamentals_backup_$(date +%Y%m%d)`는 세 요소로 구성된다. `cp -r`은 폴더 전체를 재귀적(recursive)으로 복사하는 명령, `.`과 `../`는 각각 현재 디렉토리와 상위 디렉토리를 가리키는 상대경로 표기, `$(date +%Y%m%d)`는 커맨드 치환(command substitution) 문법으로 `date` 명령의 실행 결과(예: `20260715`)를 그 자리에 텍스트로 끼워 넣는 방식이다.

### 적용

`fundamentals` 폴더 전체를 한 단계 위 경로에 `fundamentals_backup_20260715`라는 이름으로 복사해, `filter-repo`나 `merge` 작업이 잘못됐을 때 되돌릴 수 있는 안전판을 마련함.

## 트러블 슈팅 14 - `git subtree add` 명령어 분해

### 배경

TIL 저장소를 fundamentals의 하위 폴더로 병합하는 방법으로 `git subtree`가 제안되어, 동작 원리를 이해할 필요가 있었음.

### 개념 설명

`git subtree add --prefix=대상경로 소스경로 브랜치명`은 외부 저장소의 지정 브랜치 히스토리 전체를, 현재 저장소의 특정 하위 폴더(`--prefix`) 아래로 경로를 재구성해 하나의 병합 커밋(merge commit)으로 이어붙이는 Git 내장 확장 기능이다. `filter-repo` + `merge` 조합과 달리, 경로 이름을 바꾸지 않고 통째로 가져올 때 더 간단하게 쓸 수 있다.

### 적용

TIL 병합 자체는 이후 스코프 오인(트러블 슈팅 4)이 확인되며 보류되어, 실제 실행까지는 이어지지 않음.

## 트러블 슈팅 15 - `fetch`와 `merge`의 역할 차이

### 배경

`git remote add` → `fetch` → `merge` 순서로 작업하던 중, "이렇게 하면 로컬은 업데이트가 안 되는 건가"라는 의문이 생김.

### 개념 설명

`git fetch`는 원격(remote)의 커밋 데이터를 `.git` 내부 데이터베이스로 받아오기만 하는 단계로, 이 시점엔 눈에 보이는 파일(working directory)에 아무 변화가 없다. 반면 `git merge`는 fetch로 받아둔 히스토리를 현재 브랜치에 실제로 반영하는 단계로, 이 시점에 비로소 폴더와 파일이 실제로 생성·변경된다.

### 적용

`fetch`만 실행하고 결과가 안 보인다고 오해하지 않도록, 두 단계가 분리되어 있다는 전제로 매 병합 작업의 검증 시점을 `merge` 이후로 고정함.

## 트러블 슈팅 16 - `remote add → fetch → merge → remote remove` 4단계 동작 원리

### 배경

서로 다른 두 저장소(`fundamentals`, `00_BASIC_temp`)를 병합하는 네 줄짜리 명령어 블록이 반복적으로 쓰였는데, 각 단계가 정확히 무엇을 하는지 확인이 필요했음.

### 개념 설명

`remote add`는 대상 경로에 임시 이름표(예: `basic-src`)를 등록만 하는 단계, `fetch`는 그 이름표가 가리키는 저장소의 데이터를 받아 `basic-src/브랜치명`이라는 참조로 저장하는 단계, `merge --allow-unrelated-histories`는 받아둔 히스토리를 현재 브랜치에 실제로 합치는 단계, `remote remove`는 병합이 끝난 뒤 더 이상 필요 없는 이름표를 정리하는 단계다. `--allow-unrelated-histories`가 필요한 이유는, 두 저장소가 공통 조상 커밋이 없는 완전히 별개의 히스토리이기 때문에 Git이 기본적으로 병합을 거부하는 걸 명시적으로 허용하기 위함이다.

### 적용

python/java/javascript, algorithm, math 세 단위 모두 이 4단계 패턴을 그대로 반복 적용해 병합을 완료함.
