## 특강 - AI/Cloud 시대에 알아두면 쓸모있는 네트워크 인프라 이야기

- 신입 개발자를 위한 네트워크 인프라 세미나
- 내가 URL을 접속하면 어떤 일이 벌어질지 아는 대로 설명하라.

1. 인프라는 H/W에서부터 시작
   - 네트워크 장비 + 서버 + 스토리지 + 보안장비의 집합
   - 에 덧붙여 전력 + 랙 + 케이블까지
   - 앱은 이런 환경 위에서 동작

2. 네트워크 톨로지
   - 폰 -> 공유기(LAN) -> 통신사(ISP) -> 백본/인터넷
     - IDC A
     - IDC B
     - 해외 리전(해저케이블) -> 글로벌 광케이블 연결도 (Submarine Cable Map 전 세계 해저케이블 지도(TeleGeography))
       - https://www.submarinecablemap.com/

- 장애는 코드 한 줄로 발생한다. (장애가 이렇게 위험하다.)
- 국가 백본을 타고 넘어오면, 그 다음은 IDC
  - IDC(Internet Data Center): 서버, 네트워크 장비를 모아둔 시설
  - 회사는 상용 IDC에 "입주"하거나, 자체 IDC를 구축해 입점
    - 전력, 냉방, 보안, 회선이 이중화된 건물에 서버가 산다.

- 네트워크 설정까지
  - 장비를 꽂았다고 끝이 아니라, 설정이 있어야 통신이 된다.
    - VLAN, IP, 라우팅 - 한 줄 한 줄이 서비스에 직결
  - 이런게 짧게는 수백줄에서 많게는 수만줄
    - 장비가 수천대는 훌쩍
    - Meta, Google, AWS

- International Network
  - 표준 규약
    - RFC 791 - Internet Protocol
    - RFC 793 - Transmission Control Protocol

- 한국은 세계 두 번째로 인터넷 구축
  - ETRI (https://www.etri.re.kr/)
  - 나는 상대바을 어떻게 찾아 대화할 것인가, IP를 알아야 한다.
  - IP가 같으면 장애가 난다.

- IPv4는 4옥텟으로 이루어진 32비트 주소체계
  - 1옥텟 = 8비트
  - 8비트 = 256가지, 컴퓨터는 0부터 세니까 0~255

- IPv6도 있다.
  - 16bit 씩 8개 그룹

- 0.0.0.0, 255.255.255.255 특수
  - 169.254.0.1: 인터넷이 안 될 때,
  - 192.168.0.1: ~

- 미리 예약된 주소 체계
  - 모든 IP 주소엔 주인이 있음.
  - 누가 관리? IANA(Internet Assigned Numbers Authority) (https://www.iana.org)

- 실제 할당 내역
  - AWS는 얼만큼의 IP를 갖고 있는가?
  - 자사가 쓰는 공인 IP 대역 목록을 공개
  - 수천만 ~ 1억 개 이상으로 추정되는 보유량
  - 왜 이렇게 많이 샀을까?
    - MIT의 18.0.0.0/8 일부 매입
    - GE의 3.0.0.0/8 매입 (2018)
    - ARDC의 44.192.0.0/10 매입 (2019)
    - IPv4 가격 급등으로 보유 가치만 수만 달러

- 결국 인터넷 주소는 "공인"이 전부
  - "사설"은 인터넷으로 나올 수 없다.

- 사설과 공인 IP의 구분
  - RFC 1597: Address Allocation for Private Internets
  - RFC 1627 - Network 10 Considered Harmful
  - 기업의 합병, VPN, 멀티클라우드에서 겪는 RFC 1917
  - RFC 1814
  - RFC 1918 - 공식 모범 사례 정착 (논쟁의 종착점)
    - 기업의 사설 IP는 아래 3가지를 벗어나면 안 된다.
      - Class A, B, C

- Unique한 주소가 있어야 누가 누군지 알고 대화해야 한다.
  - 고유한 IP만 사용, 사설 IP를 공용으로 사용
  - IP가 부족하면 나도 갖고 싶어짐.

- 현 IPv4 할당 종료
  - 2011 방통위 보도 자료
  - https://www.korea.kr/briefing/pressReleaseView.do?newsId=155744124

- NAT를 이용해 IP 고갈 문제 해결
  - RFC 1631, NAT
  - CIDR
  - 사설망과 인터넷의 경계에 NAT 장비를 둠

- 사설 IP가 인터넷으로 나오게 되면?
  - 충돌 발생 -> 라우팅 이슈
    - 내 PC -> 기본 게이트웨이 -> 회사, 통신사 라우터 -> 중간 네트워크들 -> 8.8.8.8 네트워크
    - 각 라우터는 목적지 IP를 보고 라우팅 테이블에서 "가장 적절한 다음 경로(Next-hop)"를 찾아 전달 = Hope-by-Hope
  - 어떻게 방지?
  - 정적(static) 라우팅과 동적(Dynamic) 라우팅
    - 관리자가 직접 설정 - 정적

- 결국 우리 회사의 '코어 네트워크' 장비에서 라우팅 필터를 통해 회사 내부에서 사용하는 사설 IP가 외부로 나가는 것을 차단한다.

- 라우팅, 스위칭은?
  - 라우팅은 IP 주소를 본다.
  - 스위칭은 MAC 주소를 본다.
  - OSI 7 Layer: 네트워크 통신 과정
    - 네트워크에서 주로 2, 3, 4, 7 Layer를 자주 보게 됨.

- 실제 통신에 관여하는 구간은 Layer 2

- 어디로 보낼지 모르기 때문에 패킷이 드랍된다. 그래서 라우팅 정책이 중요하다.

- 전통적으로 3-Tier
- 현대적으로 CLOS, Leaf-Spine
  - East-West

## 2교시 - 서비스를 떠받치는 네트워크 인프라

- 로드밸런서(부하를 관리)
  - 서버가 하나의 IP로 요청을 처리
  - 사용자 증가 -> 부하 증가 -> 응답 지연 -> 서버 다운
  - Scale-out 방식이 필요
  - 사용자의 요청을 3대 중 어디로 보내야 하는가?
  - 초기엔 DNS로 분산 - DNS RoundRobin(하나로 수렴)
  - DNS가 응답 순서를 바꾸면서 사용자를 여러 서버로 나누어 보냄.
  - RFC
  - dig test.example.com

- 왜 전용 장비 생김? - DNS 한계
  - DNS 응답이 캐시되어 분배가 균일하지 않음
  - 서버가 죽어도 캐시된 주소로 접속 가능
  - 현재 서버 연결 수나 응답 속도를 알 수 없음
  - 요청 단위가 아니라 DNS 조회 단위로만 분산
  - 장애 전환이 느림

- 알아서 분배 -> 전용 로드밸런서의 등장
  - VIP(Virtual IP)
  - 연결 상태 추적
  - 자동으로 분배

- 왜 NAT 사용?
  - IP 부족함
  - NAT의 방향을 뒤집으면 LB

- Source NAT vs ~ NAT

- 초기 로드밸런서는 L4 장비
  - 출발지 IP, 목적지 IP
  - TCP, UDP 포트
  - 프로토콜
  - 현재 연결 수
- 연결을 분산하는 주요 알고리즘
  - Round Robin
  - Weighted Round Robin
  - Least Connections
  - Fastest Response
  - Source IP Hash: 출발지 IP 기준 고정 매핑

- 단순 분산에서 "장애 감지"로 발전

- Health Check - 실패하면 분배 대상에서 제외

- DNS와 로드밸런서의 장애 전환 차이
- L7 로드 밸런서
  - 추가로 필요한 기능
  - SSL/TLS 종료 (주로 이 4개)
  - HTTP 헤더 변경 (주로 이 4개)
  - 세션 지속성 (Sticky Session) (주로 이 4개)
  - 콘텐츠 기반 라우팅 (주로 이 4개)
  - 압축과 캐싱
  - WAF
  - DDoS
  - 인증, 접근 제어

- DNS는 서비스 이름과 실제 서버 위치를 분리하는 것

- 어떻게 같은 IP로 서비스를 하는가?
  - 여기서 쓰이는 게 Anycast 방식
  - 응답하는 위치가 분산
  - 동일한 IP를 전 세계 여러 곳에서 BGP로 광고

- GSLB
  - 부하 분산은 동일하지만, 글로벌하게
  - 서비스 규모가 커지면서 같은 서비스를 여러 지역에 복제하기 시작
    - 사용자를 어디로 보낼 것인가? GSLB 필요
  - GSLB = DNS + LoadBalancer

## 3교시 - 코딩테스트 해설

- 디테일을 챙기는 방법: 오답노트 정리

1. 펙토리얼
   - 완전 탐색 X -> 필요한 것만 골라서 탐색 -> 근데 왜 틀림?
   - 정수론적으로 접근해야 한다.(?)

2. 주사위 + 휴식
   - 경우의 수: 완전 탐색, DP
   - 좌, 우 두 개의 방향으로 제약을 두면 DP가 가능
   - 겹치는 부분은 메모이제이션

3. 외계인 미사일
   - 스위핑(sweeping)
   - 우선순위 큐

4. 잡초 제거
   - Update + Range Sum Query
   - Sparse Table
   - Sqrt Decomposition
   - Segment Tree

5. Segment Tree
   - 이진 트리 형태를 차용해 구간 표현, 깊이 O(log N)
   - 재귀 구현

6. Fenwick Tree (BIT)
   - 세그먼트 트리의 변형 형태
   - 역원을 가지는 연산만 사용 가능: Prefix Sum 마냥, [l, r] 구간 쿼리로 처리하기 때문
   - 구현이 간단
   - Segtree보다 시간/공간복잡도 면에서 우위
     - 시간복잡도 O(log N)으로 같지만, 비트 연산이라 상수가 작습니다.
     - 공간 또한 세그트리 4N, 펜윅트리 N으로 더 적게 차지함.

7. Sqrt Decomposition (제곱근 분할법, 평방분할법)
   - N개의 구간을 N/B (B: Bucket Size) 개의 구간으로 분할
   - i번째 원소는 floor(i/B) 구간에 포함
   - Update O(1) / Query O(N/B+B)
   - B를 sqrt(N)으로 잡으면 둘 다 O(sqrtN) 정도가 걸림.
   - 세그먼트 트리보다 구현은 대체로 간편하다.
   - N이 커질수록 O(NsqrtN)이 O(NlogN)보다 느리다.

- 케빈
  - 프로젝트의 깊이감을 얘기할 때, 단편적으로 CS 지식과 엮어서 설명하는 게 아님.
    - 깊이감 있는 문제를 설정해야 한다. (예시: 동시성 문제)
    - 다양한 방법을 제시해야 한다.
    - 방법 선택에는 이유가 있어야 한다. (시급성)
    - 동료 개발자와의 협업을 위해 개선할 사항이 무엇인지 확인한다.
    - 해당 작업으로 문제 해결 방식을 확장한다.
    - 실제로 동작하도록 구현한다.
  - 예시
    - 동시성 문제가 발생했고, 해당 문제를 해결하는 여러 가지 방법 중 가벼운 OOO의 방식을 선택해서 사용했다. 그 이유는 OO 때문이었다. 하지만, 실제 이것은 현재 방식이었고, 이후 확장성을 고려하여 더 개선해나가기 위해 XXX 작업을 추가적으로 진행해서, 다른 개발자와 협업을 위해 좀 더 개선해 나갈 수 있는 방법에 대해 고민했고, 실제로 제대로 동작했다.
      - 문제 발생
      - 원인 분석
      - 여러 가지 해결 방안
      - 그 중 OO라는 방법을 선택한 이유
      - 결론 및 시사점 (그 이후 확장성까지 고려했다는 점)
