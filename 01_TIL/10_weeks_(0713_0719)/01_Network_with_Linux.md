# Network with Linux

## 배경 및 필요성

- 월요일에 배운 Linux는 단일 시스템 내부에서 file, process, text를 다루는 도구입니다.네트워크는 이 시스템이 서로 연결되는 방식을 다룹니다. 실무에서 실제로 시스템을 운영하려면 이 둘이 항상 함께 작동합니다. 그래서 서버 한 대만 잘 다뤄서는 서비스가 안 되고, 그 서버가 네트워크로 연결되어야 합니다.

## 연결 지점

- Node, Router, Gateway: SSH는 원격 컴퓨터에 접속한다고 배웠습니다. 그 원격 컴퓨터가 네트워크상 하나듸 노드이고, 거기 도달하기까지 게이트웨이, 라우터를 거칩니다.
- Port, NAT, Port forwarding: Curl, Wget는 서버 통신 시 IP, 포트 번호가 함께 필요합니다. 기존에 "서버 통신"으로 뭉뚱그렸던 내용에서 포트 개념을 배우면 URL의 8080이 무엇을 의미하는지 이해할 수 있습니다.
- 네트워크 장비 전반: Nslookup, Dig에서 "DNS 내 도메인 이름/IP 조회"한다고 정의했습니다. 노드, IP 체계를 배우면 DNS가 왜 필요한지(사람은 IP를 못 외우니까 이름으로 노드를 검색) 원리를 채울 수 있습니다.
- 트래픽 처리, Node: Ping는 ICMP 패킷으로 "응답 확인"한다고 배웠습니다. 노드 개념을 배우면 ping이 정확이 어떤 노드 사이의 왕복을 확인하는 건지 구조가 보입니다.
- Port, Socket: SS(Socket Statistics)가 조회하는 "네트워크 연결 정보"가 포트, IP의 조합(소켓)입니다.
- Inline, Proxy: Pipe는 "표준 인터페이스 규약(stdin/stdout/stderr)으로 데이터 스트림을 연결"한다고 배웠습니다. 네트워크의 Inline 트래픽 처리(장비가 통신 경로 중간에 끼어 스트림을 그대로 통과시키는 방식)와 구조적으로 유사합니다.

## 1단계: 기초 개념

### 배경 및 필요성

- `ssh user@192.168.1.10` 명령어로 원격 컴퓨터에 접속이 가능합니다. 이 때, `192.158.1.10`와 같은 값이 무엇을 가리키는지, `ping 192.168.1.10`를 입력했을 때 오는 응답이 구체적으로 어떤 대상 사이의 왕복인지 다룰 필요가 있습니다.
- 정확히 설명하려면 "네트워크에 연결되어 통신 가능한 개별 장치"를 가리키는 용어가 필요하며, 이를 node(노드)라고 부릅니다. SSH, Ping이 공통으로 다루는 대상이 노드입니다. 두 명령어의 인자(argument)로 들어가는 IP 주소가 해당 노드를 식별합니다.

### 동작 원리

SSH 관점
`ssh user@192.168.1.10` 실행 시:

1. 로컬 컴퓨터(나의 노드)가 `192.168.1.10`이라는 노드를 목적지로 지정합니다.
2. 네트워크를 통해 해당 노드에 연결 요청을 전송합니다.
3. 목적지 노드가 응답하면 인증 절차(Secure) 후 원격 셀(Shell) 세션이 시작됩니다.

SSH는 "노드 A가 노드 B에 접속"하는 동작이고, Shell은 접속 이후 해당 노드 내부에서 실행되는 인터페이스입니다.

Ping 관점
`ping 192.168.1.10` 실행 시:

1. 내 노드가 목적지 노드로 ICMP 패킷 전송
2. 목적지 노드가 수신 즉시 응답 패킷 반송
3. 왕복 시간(round-trip time)을 측정

Ping은 "노드 간 연결이 실제로 살아있는지"를 확인하는 동작이고, SSH 접속이 실패했을 때 "노드까지는 도달하는지 아니면 노드가 아예 죽었는지"를 구분하는 진단 도구로 사용됩니다.

### 개념 정리 및 핵심 원리

- 네트워크(Network): 둘 이상의 노드가 통신 매체(유선/무선)로 연결되어 데이터를 주고받는 시스템. node(통신 주체) + link(연결 경로) + protocol(통신 규약)으로 구성됨.
- 노드(node): 네트워크에 연결되어 데이터를 송수신할 수 있는 개별 장치, 컴퓨터, 서버, 라우터, 스위치 등이 모두 해당하며, "장치 종류"가 아니라 "네트워크 상 위치를 가진 통신 주체"라는 역할(role) 개념임.
- DNS와의 연결: Nslookup, Dig로 조회하던 "도메인 이름 -> IP 주소" 변환도 결국 사람이 읽을 수 있는 이름을 "노드의 식별 주소"로 바꾸는 과정임. SSH에 IP 대신 도메인을 쓸 수 있는 이유도 이 변환이 내부적으로 일어나기 때문임.

### 결론 및 한계

이 단계에서는 "노드가 통신 주체"라는 역할만 정의하고, 노드가 물리적으로 어떤 장비인지(컴퓨터인지, 라우터인지)는 다루지 않음. 이는 3단계(네트워크 장비)에서 구체화함. 또한 노드가 "어느 범위(LAN/WAN 등) 안에 속하는지"도 현재 단계에서 제외함.

## 2단계: 네트워크 범위 (LAN / WLAN / VLAN / MAN / WAN / PAN)

### 배경 및 필요성

- 1단계에서 노드와 네트워크를 정의했는데, 실제로는 노드가 몇 개 모여있는지, 얼마나 넓은 범위에 흩어져 있는지, 유선인지 무선인지에 따라 네트워크를 다르게 설계하고 관리해야 함.
- 이 차이를 구분하는 기준이 없으면 "네트워크"라는 말이 방 안의 컴퓨터 2대를 가리키는지, 대륙 간 연결을 가리키는지 구분이 안 됨. 그래서 규모와 매체 기준으로 네트워크를 분류하는 용어들이 생김.

### 동작 원리 (분류 기준)

- 6개 용어는 하나씩 따로 외울 게 아니라, 3개의 독립적인 기준(axis) 조합으로 이해하는 게 효율적임.

기준 1: 물리적 거리(scope)
| 용어 | 범위 |
| --- | --- |
| PAN (Personal Area Network) | 개인 주변 수 미터 (예: 블루투스 이어폰-폰) |
| LAN (Local Area Network) | 건물/사무실 내부 |
| MAN (Metropolitan Area Network) | 도시 단위 |
| WAN (Wide Area Network) | 국가/대륙 단위, 인터넷이 대표적 WAN |

기준 2: 전송 매체(유선/무선)

- LAN → 유선이 기본 전제
- WLAN (Wireless LAN): LAN과 범위는 같지만 매체가 무선. 즉 WLAN은 "새로운 범위"가 아니라 "LAN의 무선 버전"임.

기준 3: 물리적 연결 vs 논리적 분리

- VLAN (Virtual LAN): 물리적으로는 같은 스위치/배선에 연결되어 있어도, 스위치 설정으로 논리적으로 별도의 LAN인 것처럼 분리하는 기술. 예: 같은 사무실 배선을 쓰지만 "개발팀 VLAN"과 "게스트 VLAN"을 소프트웨어적으로 분리.

### 결론 및 한계

이 6개 용어는 "네트워크의 크기/매체/논리 구조"를 설명하는 분류 체계일 뿐, 실제로 그 안에서 데이터가 어떻게 전달되는지는 설명하지 않음. 예를 들어 VLAN이 어떻게 논리적으로 분리되는지(태깅 방식 등)는 3단계 네트워크 장비(특히 스위치)를 배운 뒤에야 구체적으로 이해 가능함.

## 3-1단계: 네트워크 장비 - NIC, MAC 주소, 물리 계층 장비 (리피터, 허브)

### 배경 및 필요성

- 지금까지 노드를 "네트워크상 식별 가능한 통신 주체"로만 정의했는데, 실제로 노드가 되려면 물리적으로 네트워크에 연결할 하드웨어가 있어야 하고, 그 하드웨어를 유일하게 식별할 주소가 있어야 함.

### 동작 원리

NIC (Network Interface Card, 네트워크 인터페이스 카드)

- 정의: 컴퓨터를 물리적 네트워크 매체(이더넷 케이블, 와이파이)에 연결하는 하드웨어. 노드가 네트워크에 참여하려면 반드시 하나 이상의 NIC를 가져야 함.
- Linux 관점: ip addr 또는 ifconfig(구버전) 명령으로 조회되는 eth0, wlan0 같은 인터페이스가 바로 NIC를 소프트웨어적으로 표현한 것.

MAC 주소 (Media Access Control 주소)

- 정의: 각 NIC에 제조 시점에 부여되는 고유 식별자(48비트). IP 주소가 "논리적 위치"라면 MAC 주소는 "물리적 장치 자체"를 가리킴.
- IP와의 차이: IP는 네트워크를 이동하면 바뀔 수 있지만(예: 공유기 재접속 시 DHCP로 재할당), MAC 주소는 하드웨어에 고정됨.

리피터 (Repeater)

- 정의: 신호가 케이블을 타고 이동하면서 감쇠(attenuation)되는 것을 보완하기 위해, 받은 신호를 그대로 증폭해서 재전송하는 장비.
- 계층: OSI 1계층(물리 계층). 신호의 의미를 전혀 해석하지 않고 그대로 증폭만 함.

허브 (Hub)

- 정의: 여러 노드를 물리적으로 연결하는 장비. 한 포트로 들어온 신호를 나머지 모든 포트로 그대로 복제해서 내보냄(broadcast 방식).
- 계층: 물리 계층. 리피터의 다중 포트 버전이라고 볼 수 있음 — MAC 주소를 보고 목적지를 판단하지 않고 무조건 전체 전달.
- 한계: 목적지가 아닌 노드에도 신호가 전달되므로 트래픽 낭비가 크고, 충돌(collision) 가능성이 있음. 이 한계가 다음 단계(브릿지/스위치)의 필요성으로 이어짐.

### Linux와의 연결

- SS(Socket Statistics)가 보여주는 `IP:포트 조합`은 논리적 주소이고, 그 밑단에는 항상 MAC 주소로 식별되는 물리적 NIC가 존재함. 즉 SS는 3-1단계보다 상위 계층(전송 계층)을 보여주는 도구.
- Ping이 목적지 노드까지 도달하려면, 결국 같은 물리 네트워크 안에서는 MAC 주소 기반으로 실제 프레임이 전달됨(IP-MAC 매핑은 ARP 프로토콜이 담당 — 이 부분은 참고로만 언급, 오늘 키워드 범위 밖).

### 결론 및 한계

- 허브는 "목적지를 구분하지 않고 전체 전달"하는 방식이라 비효율적임. 이 문제를 해결하려고 MAC 주소를 보고 필요한 포트로만 전달하는 장비가 등장하는데, 이게 3-2단계의 브릿지/스위치임.

## 3-2단계: 네트워크 장비 - 브릿지, 스위치 (데이터링크 계층)

### 배경 및 필요성

- 3-1단계에서 허브의 한계를 짚었음. 목적지를 구분하지 않고 모든 포트로 신호를 복제 전송하기 때문에, 노드 수가 늘어날수록 불필요한 트래픽과 충돌(collision)이 증가한다는 점임. 이 문제를 해결하려면 "신호를 받았을 때 목적지가 어디인지 판단해서 필요한 곳으로만 전달"하는 장비가 필요함.
- 이걸 가능하게 하는 게 3-1단계에서 배운 MAC 주소를 읽는 능력.

### 동작 원리

브릿지 (Bridge)

- 정의: 두 개의 네트워크 세그먼트(segment)를 연결하면서, 프레임(frame)의 목적지 MAC 주소를 보고 그 프레임을 반대편 세그먼트로 넘길지 말지 판단하는 장비.
- 계층: OSI 2계층(데이터링크 계층). 허브와 달리 MAC 주소를 해석함.
- 동작: 각 세그먼트에 어떤 MAC 주소들이 있는지 학습(MAC 주소 테이블)해서, 같은 세그먼트 내부 통신이면 반대편으로 넘기지 않고 차단(filtering)함. 이게 허브 대비 트래픽을 줄이는 핵심.
- 포트 수: 일반적으로 포트 수가 적음(2~4개 수준). 세그먼트 간 연결이 목적.

스위치 (Switch)

- 정의: 브릿지와 동일한 원리(MAC 주소 기반 판단)로 동작하지만, 포트 수가 많고(수십 개) 각 포트를 사실상 독립된 세그먼트처럼 취급하는 장비.
- 관계: 스위치는 "다중 포트 브릿지"로 이해하면 됨. 원리는 같고 규모와 포트 밀도가 다름.
- 동작: 포트별로 연결된 노드의 MAC 주소를 학습(MAC 주소 테이블)해서, 프레임이 들어오면 목적지 MAC이 연결된 포트로만 전달함. 허브처럼 전체 브로드캐스트하지 않음.
- VLAN과의 연결: 2단계에서 배운 VLAN이 바로 스위치 기능으로 구현됨.

허브 vs 브릿지/스위치 비교
| 구분 | 허브 | 브릿지/스위치 |
| --- | --- | --- |
| 계층 | 물리 계층(1계층) | 데이터링크 계층(2계층) |
| 판단 기준 | 없음(무조건 전체 전달) | 목적지 MAC 주소 |
| 트래픽 효율 | 낮음(불필요한 전달 다수) | 높음(필요한 포트로만 전달) |
| 충돌 도메인 | 전체가 하나의 충돌 도메인 | 포트별로 분리됨 |

### Linux와의 연결

- `arp -a` 또는 `ip neigh` 명령으로 조회되는 ARP 테이블이, 스위치가 내부적으로 관리하는 MAC 주소 테이블과 유사한 개념 구조임. 둘 다 "어떤 주소가 어디에 있는지"를 캐시(cache)해두는 방식.
- "MAC 주소 기반 학습·캐싱"이라는 원리가 스위치와 리눅스 네트워크 스택 양쪽에서 반복되는 패턴이라는 점만 짚고 넘어감. (학습 범위 밖)

### 결론 및 한계

브릿지/스위치는 MAC 주소(2계층 주소)만 보고 판단함 — 즉 같은 네트워크 범위(같은 LAN) 안에서만 유효한 판단 방식임. 서로 다른 네트워크(예: 우리 집 LAN과 인터넷)를 연결하려면 IP 주소(3계층) 기준으로 판단하는 장비가 필요한데, 이게 3-3단계의 라우터/게이트웨이임.

## 3-3단계: 라우터, 게이트웨이 (네트워크 계층)

### 배경 및 필요성

- 3-2단계에서 브릿지/스위치의 한계를 짚었음. MAC 주소(2계층)만 보고 판단하기 때문에 같은 네트워크(같은 LAN) 안에서만 유효함. 하지만 실제로는 우리 집 LAN에서 인터넷(WAN)에 있는 서버로 나가야 하는 경우가 대부분임. 서로 다른 네트워크를 연결하고, "이 목적지가 내 네트워크 안에 있는지 밖에 있는지" 판단해서 적절한 경로로 보내는 장비가 필요함.
- 이걸 위해 MAC 주소보다 상위 개념인 IP 주소(3계층)를 읽고 판단하는 장비가 등장함.

### 동작 원리

라우터 (Router)

- 정의: 서로 다른 네트워크(예: LAN과 WAN, 또는 LAN과 다른 LAN)를 연결하고, 목적지 IP 주소를 보고 어느 경로로 패킷을 보낼지 결정(routing)하는 장비.
- 계층: OSI 3계층(네트워크 계층). 스위치가 MAC 주소 테이블을 쓰듯, 라우터는 라우팅 테이블(routing table)을 참조해서 "이 목적지 IP 대역은 어느 방향으로 보내야 하는지" 판단.
- 핵심 차이(스위치 대비): 스위치는 "같은 네트워크 안에서 어느 포트로 보낼지"를 판단하고, 라우터는 "이 패킷이 어느 네트워크로 나가야 하는지"를 판단함. 즉 스위치가 LAN 내부용이라면 라우터는 네트워크 간 경계용.

게이트웨이 (Gateway)

- 정의: 한 네트워크에서 외부(다른 네트워크, 보통 인터넷)로 나가는 출입구 역할을 하는 노드. "게이트웨이"는 특정 장비의 고유 명칭이 아니라 역할(role)임. 실제로는 대부분 라우터가 게이트웨이 역할을 겸함.
- 가정용 공유기가 대표적 예: 집 LAN(192.168.1.x) 안의 노드들이 인터넷으로 나갈 때 거치는 첫 홉(hop)이 공유기이고, 이 공유기가 라우터이자 동시에 게이트웨이임.
- "기본 게이트웨이(default gateway)": 목적지가 자신의 네트워크 대역에 없을 때, 노드가 패킷을 무조건 보내는 주소. Linux에서 ip route 명령으로 조회되는 항목이 바로 이 값.

라우터 vs 게이트웨이 관계 정리

- 라우터와 게이트웨이는 별개 장비가 아니라, 같은 장비를 어느 관점에서 부르느냐의 차이임.
  - "패킷을 어느 경로로 보낼지 결정하는 기능" 관점 → 라우터
  - "네트워크 밖으로 나가는 출입구" 관점 → 게이트웨이

3단계 전체 정리 (물리 → 데이터링크 → 네트워크)
| 계층 | 장비 | 판단 기준 | 역할 |
| --- | --- | --- | --- |
| 1(물리) | 리피터, 허브 | 없음 | 신호 증폭/단순 복제 |
| 2(데이터링크) | 브릿지, 스위치 | MAC 주소 | 같은 네트워크 내 목적지 포트 판단 |
| 3(네트워크) | 라우터, 게이트웨이 | IP 주소 | 네트워크 간 경로 판단, 외부 출입구 |

### Linux와의 연결

- "게이트웨이, 라우터를 거쳐 도달한다"고 1단계에서 언급했던 부분이 여기서 구체화됨. SSH로 원격 접속할 때, 목적지가 같은 LAN이 아니라면 반드시 기본 게이트웨이(공유기 등)를 거쳐 라우팅됨.
- `ip route` (또는 구버전 `route -n`) 명령으로 현재 노드의 기본 게이트웨이와 라우팅 테이블을 직접 확인 가능함. 학습한 "라우터가 라우팅 테이블을 참조한다"는 개념을 Linux 노드 입장에서도 동일하게 확인할 수 있는 지점.
- Nslookup/Dig로 도메인을 IP로 바꾼 다음, 실제로 그 IP까지 도달하는 경로가 바로 이 라우터들을 거치는 과정임. `traceroute` 명령으로 그 경로의 각 라우터를 순서대로 확인 가능.

### 결론 및 한계

- 3단계에서 물리 계층부터 네트워크 계층까지 장비를 순서대로 봤음. 이 장비들은 "데이터를 어떻게 전달할지"를 다루지만, 그 전달 경로 중간에서 트래픽을 감시하거나 대신 처리하는 방식(보안 장비, 프록시 등)은 다루지 않았음. 이게 4단계(트래픽 처리 방식: Inline, Out of Path, Proxy)의 주제.

## 4단계: 트래픽 처리 방식 (Inline, Out of Path, Proxy)

### 배경 및 필요성

- 3단계에서 배운 장비(스위치, 라우터 등)는 "데이터를 목적지까지 전달하는 것" 자체가 목적임. 하지만 실무에서는 전달 과정에서 트래픽을 검사하거나(보안 장비), 대신 처리하는(캐싱, 접근 제어) 장비를 추가로 배치하는 경우가 많음.
- 이때 그 장비를 "통신 경로의 어디에 두느냐"에 따라 전체 시스템의 동작 방식과 장애 영향 범위가 달라짐. 이 배치 방식을 구분하는 개념이 Inline과 Out of Path이고, 그 응용 형태 중 하나가 Proxy임.

### 동작 원리

Inline

- 정의: 트래픽 처리 장비를 통신 경로 **중간에 직접 삽입**해서, 모든 트래픽이 반드시 그 장비를 통과하도록 배치하는 방식.
- 구조: 노드 A → [Inline 장비] → 노드 B. 모든 패킷이 물리적/논리적으로 이 장비를 거쳐야만 목적지에 도달함.
- 장점: 모든 트래픽을 빠짐없이 검사·제어 가능(예: 방화벽이 악성 트래픽을 즉시 차단).
- 단점: 이 장비가 장애나면 전체 통신 경로가 끊김(단일 장애점, single point of failure). 지연(latency)도 이 장비를 거치는 만큼 항상 추가됨.

Out of Path (Out-of-Band라고도 함)

- 정의: 트래픽 처리 장비를 주 통신 경로에 직접 넣지 않고, 경로 밖에서 **트래픽의 복사본**을 받아 분석만 하는 방식.
- 구조: 노드 A → 노드 B (직접 통신). 스위치 등이 이 트래픽을 복제(mirroring)해서 별도로 [Out of Path 장비]에 전달.
- 장점: 분석 장비가 장애나도 원래 통신에는 영향 없음(가용성 높음).
- 단점: 실시간 차단이 불가능함 — 이미 원본은 지나간 뒤 복사본만 보고 있으므로, 문제를 발견해도 "사후 대응"만 가능(로그 기록, 경고 발생 등).

Inline vs Out of Path 비교
| 구분 | Inline | Out of Path |
| --- | --- | --- |
| 위치 | 경로 중간 | 경로 밖(복사본만 수신) |
| 실시간 차단 | 가능 | 불가능(사후 대응만) |
| 장애 영향 | 전체 경로 차단 위험 | 원 통신에 영향 없음 |
| 대표 예시 | 인라인 방화벽, IPS | 네트워크 탭(TAP), IDS(모니터링 전용) |

Proxy

- 정의: 클라이언트와 서버 사이에서 요청과 응답을 대신 중계하는 노드. 클라이언트 입장에서는 "서버와 직접 통신한다"고 생각하지만, 실제로는 Proxy가 요청을 받아 서버로 대신 전달하고, 서버 응답을 다시 클라이언트에게 전달함.
- Inline과의 관계: Proxy는 구조적으로 Inline 방식의 한 형태임 — 트래픽이 반드시 Proxy를 거치도록 경로에 삽입되기 때문. 다만 Proxy는 요청을 종단(terminate)했다가 새로 만들어서 다시 보낸다는 점이 다른 Inline 장비(방화벽 등)와 다름.
- 종류(참고): Forward Proxy(클라이언트를 대신해서 외부로 나감, 예: 사내망에서 인터넷 접속 시 경유) / Reverse Proxy(서버를 대신해서 외부 요청을 받음, 예: 로드밸런서).

### Linux와의 연결

- Curl: Curl을 "서버 통신" 도구로 정리했는데, 실무에서 `curl --proxy http://proxy-server:8080 https://target.com` 형태로 Proxy를 경유해서 요청을 보내는 경우가 흔함. 이때 curl 입장에서는 목적지가 target.com이지만 실제 첫 홉은 Proxy 서버가 됨. 또한, Curl가 "IP+포트로 서버와 통신"한다고만 했는데, 그 서버가 최종 목적지가 아니라 중계자(Proxy)일 수도 있다는 점이 오늘 추가되는 내용.
- SS: Proxy를 경유하는 경우, ss 명령으로 연결 상태를 보면 클라이언트-Proxy 간 연결과 Proxy-서버 간 연결이 별도의 두 소켓으로 각각 존재함. 하나의 논리적 통신처럼 보여도 실제로는 두 구간으로 분리되어 있다는 걸 확인 가능.

### 결론 및 한계

- 4단계는 "트래픽을 어디서, 어떻게 가로채거나 중계하는가"를 다뤘음. 아직 다루지 않은 건 "주소 자체를 바꾸는" 동작. 예를 들어 사설 IP를 쓰는 노드가 어떻게 공인 인터넷과 통신하는지(NAT), 포트를 외부에 노출하는 방법(Port Forwarding) 등. 이게 마지막 5단계의 주제.

## 5단계: 주소 체계 및 변환 (포트, Broadcast, NAT, Symmetric/Cone NAT, Port Forwarding, VPN)

### 배경 및 필요성

- 1단계에서 IP 주소로 노드를 식별한다고 배웠지만, 실제로는 한 노드(예: 서버 한 대) 안에서 여러 서비스(웹, SSH, DB 등)가 동시에 실행됨. IP만으로는 "어느 서비스로 보낼지"까지 구분이 안 되므로, 노드 내부에서 서비스를 구분하는 추가 주소 체계가 필요하고 이게 포트임.
- 또한 1단계에서 다룬 192.168.1.10 같은 사설 IP는 인터넷(공인망)에서 직접 통신할 수 없다는 한계가 있는데, 그럼에도 우리는 매일 사설 IP를 쓰는 노트북으로 인터넷에 접속함. 이 모순을 해결하는 게 NAT임.

### 동작 원리

포트 (Port)

- 정의: 한 노드 안에서 실행 중인 특정 프로세스(서비스)를 식별하는 16비트 숫자(0~65535).
- IP와의 관계: IP가 "어느 노드인지"를 가리킨다면, 포트는 "그 노드 안 어느 서비스인지"를 가리킴. IP + 포트 조합을 소켓(socket)이라고 하며, 이게 앞서 SS 명령으로 조회했던 정보임.

Broadcast

- 정의: 하나의 송신자가 같은 네트워크(대개 LAN) 안의 모든 노드에게 동시에 데이터를 전송하는 방식.
- 3-1단계의 허브 동작과 원리가 같음 — 허브가 받은 신호를 모든 포트로 복제 전송했던 것처럼, Broadcast는 논리적으로 "모두에게 보낸다"는 의미의 전송 방식.
- 목적지 주소: IPv4에서는 해당 네트워크 대역의 마지막 주소(예: 192.168.1.255)를 브로드캐스트 주소로 씀.

NAT (Network Address Translation, 네트워크 주소 변환)

- 정의: 사설 IP를 쓰는 노드가 인터넷(공인망)과 통신할 때, 라우터(주로 게이트웨이 역할을 겸하는 공유기)가 사설 IP를 공인 IP로 변환해주는 기술.
- 동작: 노드가 192.168.1.10:5000으로 인터넷에 요청을 보내면, 라우터가 이를 공인IP:임의포트로 바꿔서 내보내고, 응답이 오면 다시 원래 사설 IP:포트로 되돌려줌. 이 매핑을 라우터가 테이블로 관리함.
- 3-3단계 연결: 라우터가 라우팅 테이블 참조 판단과 별개로, NAT 변환 테이블도 함께 관리하는 것. 게이트웨이가 "출입구" 역할을 한다고 했던 부분이 여기서 "주소를 바꿔서 내보낸다"는 구체적 동작으로 이어짐.

Symmetric NAT vs Cone NAT

- 공통점: 둘 다 NAT 방식의 세부 유형이며, "외부에서 들어오는 응답을 어떤 기준으로 허용할지"가 다름.
- Cone NAT: 내부 노드가 한 번 사설IP:포트 → 공인IP:포트로 매핑을 만들면, 이후 외부의 어떤 상대방이 그 공인IP:포트로 보내는 응답이든 동일하게 허용함. (Full/Restricted/Port Restricted Cone으로 세분화되지만, 범위 밖의 내용으로 제외)
- Symmetric NAT: 내부 노드가 통신하는 목적지(상대방 IP:포트)가 다르면, 매번 다른 공인 포트로 매핑을 새로 만듦. 즉 "누구와 통신하느냐"까지 매핑에 포함되어, 사전에 통신하지 않은 상대방의 응답은 차단됨.
- 실무 영향: Symmetric NAT는 보안성이 높지만, P2P 연결(예: 화상통화, 온라인 게임)에서 상대방이 접속하기 어려운 문제(NAT traversal 문제)를 일으킴.

Port Forwarding (포트 포워딩)

- 정의: NAT 뒤에 있는 사설 IP 노드의 특정 포트를, 라우터가 외부에서도 접근 가능하도록 공인 IP의 특정 포트와 고정으로 매핑해두는 설정.
- 필요성: NAT는 기본적으로 "내부 → 외부"로 나가는 통신의 응답만 받도록 설계되어 있어서, 외부에서 내부로 먼저 접속을 시작할 수 없음. 예를 들어 집에서 개인 서버를 운영하고 외부에서 SSH로 접속하려면, 라우터에 "공인IP의 2222번 포트로 들어오는 요청은 내부 192.168.1.10:22로 보내라"는 규칙을 미리 설정해야 함 — 이게 Port Forwarding.

VPN (Virtual Private Network)

- 정의: 공용망(인터넷) 위에 암호화된 가상의 사설 네트워크를 구성해서, 물리적으로 떨어진 노드를 마치 같은 사설망(LAN)에 있는 것처럼 연결하는 기술.
- 지금까지 배운 개념 종합: 사설 IP 대역(1단계) + 원격 노드 연결(3단계 라우팅) + 암호화(오늘 새로 추가되는 요소)를 결합한 응용 기술. 즉 VPN 접속 시 노드에는 가상의 사설 IP가 새로 할당되고, 모든 트래픽이 암호화된 터널(tunnel)을 통해 전달됨. 구조적으로 Inline(4단계)과 유사하게, 모든 트래픽이 VPN 터널을 반드시 거치도록 강제됨.

### Linux와의 연결

- SS: `ss -tuln` 명령으로 현재 노드에서 열려 있는 포트 목록을 직접 확인 가능함. 오늘 배운 포트 개념을 실제로 눈으로 검증하는 도구.
- Curl/Wget: 서버 통신 시 URL에 포트를 명시(:8080 등)하지 않으면 프로토콜별 기본 포트(HTTP=80, HTTPS=443)가 자동 적용됨. "IP+포트가 필요하다"고 4단계에서 언급했던 부분의 구체적 규칙.
- SSH: Port Forwarding 개념은 SSH 자체 기능(`ssh -L`, `ssh -R`)으로도 구현 가능함. 라우터 설정이 아니라 SSH 세션 안에서 로컬/원격 포트를 터널링하는 방식으로, 오늘 배운 라우터 기반 Port Forwarding과 원리는 같고 구현 주체만 다름.

### 결론 및 한계

- 5단계까지 완료하면서 전체 로드맵의 흐름을 정리하면:
  - **노드 식별(1단계) → 노드의 집합 범위(2단계) → 노드를 물리/논리적으로 연결하는 장비(3단계) → 그 경로 안에서 트래픽을 처리하는 방식(4단계) → 노드 내부 서비스 식별과 주소 변환(5단계)**으로 이어짐.
- 오늘 다룬 NAT/VPN은 "사설망과 공용망의 경계"를 다루는 개념인데, 이 경계에서 실제로 어떤 보안 위협이 발생하고 어떻게 방어하는지(방화벽 규칙, 포트 스캔 등)는 오늘 범위 밖이라 향후 보안 관련 학습에서 다뤄질 것으로 보임.
