# Trouble Shooting

## 1. directory rename에 따른 terminal AI(Claude Code)의 추적 실패 원인 분석

### 배경·필요성 — 세션이 경로를 식별하는 방법

Claude Code는 세션을 저장할 때 프로젝트 Directory의 Absolute Path String(파일 시스템 내 전체 경로)을 키(key)로 사용해서 `~/.claude/projects/` 아래에 하위 Directory를 생성합니다. (정확히는 Slash(/)를 hyphen(-)으로 치환하여 사용) 이 문자열 자체가 세션 Transcript(대화 기록 원본)를 찾는 **유일한 색인 키**입니다.

### 동작 원리 — rename이 반영되지 않는 이유

**표 1. 경로 추적 실패의 구조적 원인**
| 계층 | 동작 | rename에 대한 반응 |
| --- | --- | --- |
| Claude Code 세션 인덱스 | 세션 생성 시점에 절대 경로 **문자열**을 스냅샷(snapshot, 그 순간의 값)으로 저장 | 이후 파일 시스템 변화를 감시(watch)하지 않음 |
| 파일 시스템 (file system) | `mv 01_dev_project 99_dev_project` | inode(아이노드, 파일의 실제 저장 단위)는 그대로, 경로 문자열만 변경 |
| resume 조회 | 저장된 문자열과 현재 `cwd` (current working directory, 현재 작업 디렉터리)를 **문자열 비교**로 매칭 | 문자열이 다르면 다른 프로젝트로 간주 |

Claude Code는 Directory를 inode나 심볼릭 링크(symbolic link)가 아니라 문자열로 식별합니다. macOS 파일 시스템 관점에서 `mv 01_dev_project 99_dev_project`는 동일 inode를 가리키는 이름만 바꾸는 연산이라 논리적으로는 "같은 폴더"이지만, Claude Code의 세션 인덱스는 파일 시스템 이벤트(rename event)를 subscribe하지 않고 세션 생성 시점에 딱 한 번 경로를 기록하므로, 이후 rename이 일어나도 인덱스가 자동 갱신되지 않습니다. 이것은 "세션 = 생성 시점 경로 문자열의 스냅샷"이라는 **설계상 trade-off(상충 관계)**입니다.

이 설계를 뒷받침하는 근거는 아래 공식 문서 "세션 저장·/cd 재배치 동작"의 동작과 일치합니다. 현재 위치와 무관한 세션을 선택하면 재개하지 않고 cd 명령을 복사해주는 이유가 애초에 시스템이 "그 세션의 홈 경로"를 문자열 그대로만 알고 있기 때문입니다.

### 재발 방지 대안

**표 2. rename 이후 세션 연속성 확보 전략**
| 전략 | 방법 | 트레이드오프 |
| --- | --- | --- |
| A. 세션 이름 부여 | `claude -n "<이름>"` 또는 세션 내 `/rename <이름>` | 경로 대신 이름으로 피커 검색 → rename에 영향받지 않음. 단, 매번 이름을 붙이는 습관 필요 |
| B. 디렉터리 이름 고정 | 이후 rename을 피하거나, rename 직후 `/cd .`로 현재 경로를 재등록 | 근본적 회피지만 폴더 구조 변경 유연성 감소 |

실무적으로 A(이름 부여)가 더 견고한 방안입니다. 경로 문자열이라는 취약한 키에 의존하지 않고, 사람이 붙인 라벨로 검색하기 때문입니다.

### 결론·한계

- 결론: Claude Code의 세션 인덱스는 파일 시스템의 실시간 상태(state)를 추적하는 것이 아니라, 세션 생성 시점의 절대 경로 문자열을 정적으로 저장합니다. mv, Finder rename처럼 세션 밖에서 일어나는 경로 변경은 이 인덱스에 반영되지 않으며, /cd 명령으로 **세션 내부에서 명시적으로 이동시킨 경우만 예외적으로 재배치**됩니다.
- 한계: 이 설계의 근본 원인(왜 문자열 스냅샷 방식을 선택했는지, 즉 inode 기반 추적을 도입하지 않은 이유)은 공식 문서에 명시적으로 설명되어 있지 않습니다. 추정으로는 Cross-platform(macOS/Windows 등 여러 OS) 호환성 — inode는 macOS/Linux 개념이고 Windows에는 대응 개념이 다르므로, OS 독립적인 경로 문자열을 키로 쓰는 편이 구현이 단순하기 때문일 가능성이 높습니다.
