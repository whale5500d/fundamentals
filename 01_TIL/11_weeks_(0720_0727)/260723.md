# 목 - 심화

## AWS

- API GW: 클라이언트(웹, 모바일 등)와 백엔드 서비스(EC2) 사이에서 API 요청을 관리, 변환, 보호해주는 완전관리형 서비스
  1. (관리) 환경을 구분하여 관리. stage 기능으로 dev, test, staging, prod 4가지 환경을 분리 배포함.
  2. (관리) 로깅으로 관리, GW를 거치는 모든 요청·응답을 관측(observability). CloudWatch Logs 연동이 대표적.
  3. (보호) 인증으로 보호, 백엔드(EC2) 도달 전 게이트웨이에서 신원을 검증해 무단 요청 차단.

- Lambda: 서버를 직접 관리하지 않고 코드만 실행할 수 있는 서버리스 컴퓨팅 서비스
  - (배경) on-premise(직접 관리)의 초기 투자, 유지보수 부담을 해결하기 위해 사용한 만큼 지불하는 클라우드가 등장함. 해당 전환점이 사용자(기업)의 부담을 아래와 같이 점차 줄여줌.
    1. 온프레미스
       - 인프라 관리 책임자: 사용자(기업)
       - IDC에 랙(rack) 단위로 서버를 직접 두고 운영.
    2. 가상화
       - 인프라 관리 책임자: (일부) 클라우드 사업자
       - 물리 서버를 가상 인스턴스로 분할하여 제공.
       - 사용자 책임 = OS + 런타임 + 애플리케이션 코드 + 스케일링
    3. 서버리스
       - 인프라 관리 책임자: (전적) 클라우드 사업자 (OS, 스케일링, 인스턴스 개수까지 담당)
       - 가상 인스턴스 관리를 AWS가 담당하여, 사용자는 코드 실행에만 집중
       - 사용자 책임 = 애플리케이션 코드 + 실행 설정(Memory, Timeout, Trigger)
       - 콜드 스타트(cold start): 상시 대기 서버가 없어 요청 시 컨테이너를 초기화하며 그만큼 지연 발생. 지연에 민감하거나 실행 시간이 긴 작업은 EC2 병행.

## 배포

- 위챌 1
  - 빅뱅 배포
    1. 서버 확보
       1. EC2 인스턴스 생성
          1. SSH 22
          2. pem
    2. 서버에 접속
       1. git clone 프로젝트\_URL
       2. uv 설치
       3. uv sync (파이썬 설치, 가상 환경 생성, 의존성 설치)
       4. source .venv/bin/activate
       5. .env 설정
       6. 서버 시동

  - 도커 배포
    1. 도커 이미지 생성
       1. 도커 파일 작성
       2. docker build --platform linux/amd64 -t rag-name .
    2. 만든 도커 이미지를 도커 컴포즈를 실행
       1. docker-compose.yml 작성

- 위챌 2
  - 내가 만든 도커 이미지를 도커 허브에 올린다.
    - generate new token
    - access permissions를 반드시 write가 가능하게 변경
  - EC2에서 해당 이미지를 받아 실행한다.
    - docker compose up
    - 브라우저에서 테스트하기
