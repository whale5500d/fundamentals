### 인프라 기초 (AWS)

- EC2: 인스턴스(가상 서버)를 생성, 구성, 관리하는 AWS의 서비스
  - CPU 아키텍쳐: x64(O) vs arm64(회사에서 많이 쓴다.)
    - 스푼 SRE 30% 이상이면 아주 성공적 (스푼 사례)
    - Graviton Amazon Intel AMD

- Security Group: 하나의 인스턴스에서 트래픽을 제어하는 가상 방화벽. 인바운드 규칙을 표현
  - inbound
    - inbound 관광객(한국으로 들어오는 관광객)
    - 들어오는 트래픽
    - 기본적으로 닫혀있다.
    - 허술하게 열어두면 털림.
      - 개인정보 유출
    - 인바운드 규칙 (AWS Security Group)
      - 유형: 사용자 지정 TCP
      - 소스 유형: 사용자 지정
  - outbound
    - 나가는 트래픽
    - 기본적으로 열려있다.
  - 규칙 저장 누르기

### 컨테이너 기초

- Portainer (생략)
- Docker: 애플리케이션 실행에 필요한 모든 요소를 하나로 묶어서 어떤 컴퓨터에서든 동일하게 실행되도록 도와주는 도구
  - 보편적인 동작 및 동작 순서
    1. HW
    2. OS(Linux)
    3. Python 3.10
    4. uv sync
    5. activate
    6. FastAPI server 띄우기: `uv run uvicorn main:app --reload`
  - 도커는 리눅스 에뮬레이터(시뮬레이터)이다.
    ```
    docker run -d -p 8080:8080 --name express-app --network express-proxy-net iasnadcb/dokker-test-app:latest
    ```

- 도커 구성요소
  - 도커 파일: 도커 이미지를 생성하기 위한 청사진 역할을 하는 텍스트 문서
  - 도커 이미지: 사용자가 실행할 코드가 있는 바이너리
  - 도커 레지스트리: 도커 이미지를 저장하고 배포하는 중앙 저장소
    - 레지스트리: 데이터를 체계적으로 저장하고 관리하는 DB 또는 시스템
  - 도커 데몬: 도커 컨테이너를 관리하는 백그라운드 프로세스
  - 도커 컨테이너: 도커 이미지가 메모리 위에 상주하여 실제 코드가 수행되는 프로세스
  - 데몬: 백그라운드에서 실행되며 특정 작업을 수행하는 프로그램

- 도커 사용 이유
  - Node 백엔드를 띄우는 게 목표
  - 띄우기 위한 환경이 까다로움. (직접 관리가 어려움)
  - 빅뱅 배포는 도커를 사용하지 않음. (직접 설치, Natively - 직접 진행이라는 의미)

- 파일 이름에 확장자가 없음.

## 5교시 - 컨테이너 관리, 네트워크, 프록시

- Portainer (생략)

- Volume: 컨테이너 간 데이터를 공유 및 영속성을 제공하기 위해 호스트 파일 시스템에 생성되는 독립적인 저장소
  - 컨테이너 종료 및 재시작에도 데이터를 영구적으로 유지하고 공유하기 위해
  - 기본적으로 Docker는 Stateless(비영속적), 로그, DB, 설정 파일처럼 유지가 필요한 데이터에는 비적합
  - 컨테이너 외부에 독립적인 디스크 저장 영역을 제공하여 데이터를 영구적으로 보존하고, 여러 컨테이너에서 공유하거나 재사용할 수 있도록 함.
  - Container에는 기본적으로 상태가 없으므로, 다음 실행 간의 관계를 이어주기 위해 존재

- Bridge Network: 컨테이너들이 동일한 호스트 내에서 서로 통신할 수 있도록 연결해주는 도커의 기본 가상 네트워크
  - 볼륨(스토리지): 컨테이너 간의 데이터 공유, 상태 유지
  - 브릿지(네트웍): 컨테이너 간의 네트워크 공유, 연결
  - DB, FE, BE, (캐시 - 레디스), (메시지큐 - RabbitMQ)

- Reverse Proxy: 클라이언트의 요청을 대신 받아 서버에 전달하고, 서버의 응답을 클라이언트로 반환하는 중개 역할하는 서버
  - 중개자 역할
  - Nginx가 수행함.

- Docker Compose: 여러 컨테이너로 구성된 애플리케이션을 YAML 형식의 파일로 정의하고, 단일 명령어로 동시에 실행 및 관리할 수 있도록 지원하는 Docker의 공식 오픈소스 도구

## 6교시 - CI/CD 도구

- CI: 개발자가 작성한 코드를 자동으로 통합 및 테스트하여 코드의 품질을 지속적으로 유지하는 개발 프로세스
  - 젠킨스(Jenkins)
  - 빌드에 대한 얘기

- CD: 코드가 자동화된 테스트를 통과하면 운영 환경까지 자동으로 배포하는 배포 프로세스
  - 배포 파이프라인을 설계
  - 프론트, 백엔드: 서버에 배포
  - 앱: 스토어 배포

- 도구 (Github Actions vs Jenkins)
  - Github Actions: Github 내에서 코드 빌드, 테스트, 배포 등의 워크플로우를 자동화하는 CI/CD 도구
  - Jenkins: SW의 개발 프로세스의 빌드, 테스트, 배포 작업을 자동화하는 오픈 소스 CI/CD 도구
